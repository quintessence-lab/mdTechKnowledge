✅メリット

MERIT 01
リポジトリの構造的探索が可能

ファイルツリーを直接一覧できる。Web フェッチではカテゴリページから URL をたどる必要があるが、MCP なら「src/content/blog/ 以下のファイルを全部見せて」のような指示で即座にアクセスできる。

MERIT 02
Markdown ソースをノイズなしで取得

Web フェッチではビルド後の HTML が返り、ナビゲーション・フッター・スタイル情報が混入する。MCP なら .md の生データが取れるため、解析・要約時のノイズが少ない。

MERIT 03
リポジトリ内の横断検索

コード検索ツールにより、特定キーワードを含む記事をリポジトリ全体から探索できる。Web フェッチではサイト内検索（JS 動作）が利用できないため、MCP の優位性がある。

MERIT 04
Issue / PR 操作の統合

Claude との会話内から直接 Issue 作成や PR 作成が可能。ナレッジの改善提案を Issue 化したり、修正内容を PR として提出するワークフローが会話内で完結する。

MERIT 05
スケーラビリティ

記事が 100 件を超えてカテゴリが複雑化した場合、Web フェッチでは一覧取得に複数回アクセスが必要。MCP ならファイルツリーから一括探索でき、規模拡大に強い。

⚠️デメリット

⚠ DEMERIT 01 — 最重要
認証トークン（PAT）のセキュリティリスク

Public リポジトリであっても MCP 経由では PAT または OAuth 認証が必須。Web フェッチでは認証不要で公開 URL を読むだけだが、MCP では必ずトークンを AI に渡す必要がある。

PAT のアクセス範囲が自リポジトリに限定されない

PAT は発行時に自動的に 全 Public リポジトリへの読み取り権 を持つ。自分の mdTechKnowledge だけでなく、GitHub 上のあらゆる Public リポジトリにアクセス可能な状態で AI に渡すことになり、悪意ある Public リポジトリを通じたプロンプトインジェクション攻撃のリスクがある。

PAT スコープ設定の複雑さ

35 以上のスコープ選択肢があり、適切な判断が難しい。Classic PAT で広いスコープを設定すると、Private リポジトリや組織リソースまで AI にアクセスさせてしまう危険がある。

トークン種別アクセス制御ツール表示リスク
Classic PAT (ghp_)スコープ単位（広い）自動フィルタ広範囲・漏洩時の影響大
Fine-grained PATリポジトリ単位（狭い）全ツール表示スコープ検出非対応
OAuth認可時に選択スコープチャレンジ組織ポリシーに依存

DEMERIT 02
コンテキストの常時消費

ツール定義（スキーマ）が会話コンテキストに常時載り、数百〜1,000+ トークンを消費する。小規模サイトではこのオーバーヘッドが相対的に大きい。

DEMERIT 03
初期セットアップのコスト

PAT の発行（35+ スコープから適切な選定）、MCP サーバの登録、接続確認が必要。Web フェッチは URL 指定だけで即利用可能。

DEMERIT 04
push は Claude からできない

MCP 経由でリポジトリを「読む」ことはできるが、ファイル変更の push はできない。デプロイのトリガーは従来通り端末からの手動 push が必要。

DEMERIT 05
Web フェッチで十分なケースが多い

今回の検証で確認した通り、カテゴリページが整理されていれば Web フェッチだけで記事一覧の取得→本文読み込みが問題なく行える。

DEMERIT 06
Claude Code 前提の機能が多い

Issue 作成・PR 作成・Actions 監視などは Claude Code での利用を主に想定。claude.ai の Web チャットからはこれらの恩恵が限定的。

💡結論と推奨

現時点の推奨: Web フェッチ運用を継続

32 記事・4 カテゴリの現規模では、Web フェッチで十分対応可能。コンテキスト消費に加え、PAT の発行・管理・セキュリティリスクを考慮すると、認証不要で即利用可能な Web フェッチの方がコスト対効果が高い。

MCP 導入を検討すべきタイミング

  • 記事数が 100 件を超え、カテゴリが複雑化した場合
  • Claude との会話内から Issue / PR を直接操作したい場合
  • Markdown ソースの生データを正確に取得する必要がある場合
  • 複数リポジトリを横断して検索したい場合

MCP 導入時のセキュリティ推奨事項

  • Fine-grained PAT を使用し、対象リポジトリを明示的に限定する
  • 読み取り専用のスコープに制限する（書き込み権限を付与しない）
  • PAT の有効期限を設定し、定期的にローテーションする
  • .mcp.json を .gitignore に追加し、誤コミットを防止する
  • 環境変数で PAT を管理し、設定ファイルに直接書かない

併用という選択肢

Web フェッチと MCP は排他的ではない。通常の参照は Web フェッチで行い、構造的探索や操作が必要なときだけ MCP を利用する併用方式も現実的な選択肢である。
