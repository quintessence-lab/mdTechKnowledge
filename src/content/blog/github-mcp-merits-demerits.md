# GitHub MCP サーバ導入のメリットとデメリット

**作成日**: 2026年4月13日  
**検証方法**: Claude (claude.ai) が Web フェッチで対象サイトを実際に取得・検索し、MCP 方式との差異を評価

---

## 対象サイトの概要

mdTechKnowledge は Astro + GitHub Pages で運用されている技術ナレッジサイトである。Markdown 中心で記述されており、全 32 記事が 4 カテゴリに分類されている。

| カテゴリ | 記事数 | 主なトピック |
|---|---|---|
| Claude技術解説 | 18件 | Claude Code, MCP, Ultraplan, Agent Teams, Cowork |
| その他技術 | 9件 | Dify, Azure AI Foundry, Copilot, Vercel |
| 一般リサーチ | 3件 | 各種リサーチ |
| リーダーシップ | 2件 | リーダーシップ関連 |

---

## 2つのアクセス方式

Claude がナレッジサイトのデータを参照するには、大きく分けて 2 つの方式がある。

### 方式 A: Web フェッチ（従来方式・現在利用可能）

```
あなた ── ブラウザで閲覧
              │
Claude ── web_fetch でサイト URL を取得
              │
              ▼
     GitHub Pages（公開 HTML）
```

- Claude が公開 URL を指定して HTML を取得する
- カテゴリページ → 個別記事 URL の順にたどる
- セットアップ不要。今すぐ利用可能

### 方式 B: GitHub MCP Server 経由

```
あなた ── 端末で git push（従来通り）

Claude ── MCP Server ── GitHub API ── リポジトリ
              │
         ファイルツリー・Markdown ソースを直接取得
```

- Claude が GitHub API 経由でリポジトリ内のファイルを直接参照する
- ディレクトリ構造の一覧、任意の .md ファイルの取得が可能
- MCP 接続の初期設定が必要

## メリット

### 1. リポジトリの構造的探索が可能

MCP 経由ではファイルツリーを直接一覧できる。Web フェッチではカテゴリページやサイトマップから URL をたどる必要があるが、MCP なら「src/content/blog/ 以下のファイルを全部見せて」のような指示が可能になる。

### 2. Markdown ソースをノイズなしで取得

Web フェッチでは Astro がビルドした HTML が返るため、ナビゲーションバー、フッター、スタイル情報などが本文に混入する。MCP 経由なら `.md` ファイルの生データが取れるため、Claude が解析・要約する際のノイズが少ない。

### 3. コピペ往復が不要

従来方式でもコピペは不要（Web フェッチで読める）だが、MCP ではさらにリポジトリ内の横断検索（コード検索）が可能になる。特定のキーワードを含む記事を探す場合に有効。

### 4. Issue / PR 操作の統合

Claude との会話の中から直接 Issue の作成や PR の作成ができる。ナレッジサイトの改善提案を Issue 化したり、修正内容を PR として提出するワークフローが会話内で完結する。

### 5. 記事数が増えたときのスケーラビリティ

現在 32 記事だが、100 記事を超えてカテゴリが複雑化した場合、Web フェッチでは一覧取得に複数回のアクセスが必要になる。MCP ならファイルツリーから一括探索できるため、規模拡大に強い。

---

## デメリット

### 1. 【重要】認証トークン（PAT）のセキュリティリスク

GitHub MCP Server は **Public リポジトリであっても認証が必須** である。Web フェッチでは公開 URL をそのまま読むだけで認証不要だが、MCP では必ず Personal Access Token（PAT）または OAuth が必要になる。

これにより、以下のセキュリティリスクが発生する。

#### PAT のアクセス範囲が自リポジトリに限定されない

PAT は発行時に自動的に **全ての Public リポジトリへの読み取りアクセス権** を持つ。つまり、自分のリポジトリだけでなく、GitHub 上のあらゆる Public リポジトリにアクセス可能な状態で AI に渡すことになる。悪意ある Public リポジトリのコンテンツを通じたプロンプトインジェクション攻撃のリスクがコミュニティでも議論されている。

#### PAT スコープ設定の複雑さ

GitHub の PAT には 35 以上のスコープ選択肢があり、適切なスコープの判断が難しい。Classic PAT（`ghp_` プレフィックス）で広いスコープを設定してしまうと、Private リポジトリや組織リソースまで AI にアクセスさせてしまう危険がある。

#### トークン種別ごとのトレードオフ

| トークン種別 | アクセス制御 | ツール表示 | リスク |
|---|---|---|---|
| Classic PAT (`ghp_`) | スコープ単位（広い） | スコープに応じて自動フィルタ | 広範なアクセス権、漏洩時の影響大 |
| Fine-grained PAT (`github_pat_`) | リポジトリ単位（狭い） | 全ツール表示（フィルタなし） | スコープ検出非対応、権限エラーが実行時まで分からない |
| OAuth | 認可時に選択 | スコープチャレンジ方式 | 組織ポリシーに依存 |

#### PAT 漏洩リスク

PAT は MCP の設定ファイル（`.mcp.json` 等）や環境変数に保存される。設定ファイルを誤って Git にコミットしたり、環境変数が露出した場合、トークンが漏洩する。

#### Web フェッチとの対比

| 項目 | Web フェッチ | MCP（PAT 認証） |
|---|---|---|
| 認証 | 不要（公開 URL） | 必須（PAT or OAuth） |
| アクセス範囲 | 指定した URL のみ | PAT のスコープに応じた全リポジトリ |
| トークン管理 | なし | 発行・保管・ローテーションが必要 |
| 漏洩リスク | なし | 設定ファイル・環境変数からの漏洩リスク |
| 第三者リポジトリへのアクセス | 意図的に URL を指定した場合のみ | PAT 経由で自動的にアクセス可能 |

### 2. コンテキストの常時消費

MCP サーバを接続すると、ツール定義（スキーマ）が会話のコンテキストに常時載る。GitHub MCP Server の場合、リポジトリ操作・Issue・PR・Actions・Security などのツール群で数百〜1,000 トークン以上を常時消費する。小規模サイトでは、このオーバーヘッドが相対的に大きい。

| 消費タイミング | 内容 | 推定トークン数 |
|---|---|---|
| 常時（会話中ずっと） | ツール定義（スキーマ） | 数百〜1,000+ |
| 都度（ファイル取得時） | API レスポンス（ファイル内容） | ファイルサイズに比例 |

### 3. 初期セットアップのコスト

MCP 接続には PAT の発行（スコープの適切な選定を含む）、MCP サーバの登録設定、接続確認が必要になる。Web フェッチは URL を指定するだけで即利用できるため、導入の手軽さでは劣る。

### 4. push は Claude からできない

MCP 経由で Claude がリポジトリを「読む」ことはできるが、ファイルの変更を push して GitHub Pages を更新することはできない。デプロイのトリガーは従来通り端末からの手動 push が必要。

### 5. Web フェッチで十分なケースが多い

mdTechKnowledge はカテゴリページが整理されており、Web フェッチだけで記事一覧の取得→個別記事の本文読み込みが問題なく行える。MCP を導入しなくても「このサイトの記事を読んで」という指示に対応可能。

### 6. Claude Code 前提の機能が多い

GitHub MCP Server のツール群（Issue 作成、PR 作成、Actions 監視など）は Claude Code での利用を主に想定している。claude.ai の Web チャットからはこれらの操作系ツールの恩恵が限定的になる場合がある。

## 結論と推奨

### 現時点の推奨: Web フェッチ運用を継続

mdTechKnowledge は 32 記事・4 カテゴリの小〜中規模サイトであり、カテゴリページが整理されているため Web フェッチで十分対応できる。MCP のツール定義によるコンテキスト消費に加え、PAT の発行・管理・セキュリティリスクを考慮すると、現状では Web フェッチの方がコスト対効果が高い。

特に、Public リポジトリの参照が目的であれば、Web フェッチは認証不要で即利用可能な点が大きな優位性となる。

### MCP 導入を検討すべきタイミング

以下の条件に当てはまる場合、MCP 導入のメリットが上回る。

- 記事数が 100 件を超え、カテゴリが複雑化した場合
- Claude との会話内から Issue / PR を直接操作したい場合
- Markdown ソースの生データを正確に取得する必要がある場合
- 複数リポジトリを横断して検索したい場合

### MCP 導入時のセキュリティ推奨事項

MCP を導入する場合は、以下のセキュリティ対策を推奨する。

- Fine-grained PAT を使用し、対象リポジトリを明示的に限定する
- 読み取り専用のスコープに制限する（書き込み権限を付与しない）
- PAT の有効期限を設定し、定期的にローテーションする
- `.mcp.json` や `.mcp.local.json` を `.gitignore` に追加し、誤コミットを防止する
- 環境変数で PAT を管理し、設定ファイルに直接書かない

### 併用も有効

Web フェッチと MCP は排他的ではない。通常の参照は Web フェッチで行い、構造的な探索や操作が必要なときだけ MCP を利用する併用方式も現実的な選択肢である。

## 補足: コンテキスト消費の比較

```
【Web フェッチの場合】
会話開始時のオーバーヘッド: ほぼゼロ
記事取得時: HTML 全文 ≒ 記事本文 + ナビゲーション（やや冗長）

【MCP ありの場合】
会話開始時のオーバーヘッド: ツール定義 ≒ 数百〜1,000+ トークン（常時消費）
記事取得時: Markdown 生データ ≒ 記事本文のみ（ノイズ少）

→ 1〜2 記事の参照なら Web フェッチの方が総コンテキスト消費が少ない
→ 多数記事の横断参照では MCP の方がノイズ削減効果で逆転する可能性あり
```
