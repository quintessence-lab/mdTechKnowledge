## はじめに — 「ultracode」と書くだけで何が起きるのか

2026年5月28日（PT）、Anthropic は Claude Opus 4.8 と同時に、Claude Code v2.1.154 で **Dynamic Workflows（動的ワークフロー）** を Research Preview として公開しました。

使い方は拍子抜けするほど単純です。プロンプトのどこかに **`ultracode`** という単語を含めるだけ。するとClaude は、そのタスクをこなすための **JavaScript スクリプトを自分で書き起こし**、専用ランタイムが**バックグラウンドで多数のサブエージェントをファンアウト実行**します。

> **トリガー語の変更（v2.1.160）**: Dynamic Workflows のトリガーキーワードは、**v2.1.160（2026-06-01 PT）で `workflow` から `ultracode` へ改称**されました（[公式changelog](https://code.claude.com/docs/en/changelog)）。これにより **`workflow` という単語自体ではワークフローが起動しなくなり**ました（自然な言葉で「ワークフローを作って」と頼むのは引き続き有効）。トリガー語は入力欄で**紫（violet）にハイライト**されます。本記事は公開当初 `workflow` をトリガー語として解説していましたが、本変更を反映して `ultracode` に更新しています。同時に最大16、1回の実行で総計最大1000のエージェントが、並列にブレストし、互いの結果を敵対的にレビューし、答えが安定するまで収束していく——これが Dynamic Workflows です。

本記事は、単なる機能紹介ではなく、**「中で何が起きているのか」「どう制御するのか」「いくらかかるのか」**という実装者目線の疑問に答えることを目的とします。リリース告知レベルの概要は[Claude Code バージョン履歴まとめ](/mdTechKnowledge/blog/claude-code-version-history/)、Routines や `/code-review`（旧 `/ultrareview`）との運用は各専用記事に譲り、本記事は **Dynamic Workflows そのものの内部構造と制御**に集中します。

> 本記事は2026年5月時点の公式ドキュメント・公式ブログ・報道に基づきます。Research Preview のため、数値・挙動は今後変更され得ます。

## 1. Dynamic Workflows とは何か

### 本質：サブエージェントを束ねる「自動生成スクリプト」

Dynamic Workflow の正体は、**「サブエージェントを大規模にオーケストレーションする JavaScript スクリプト」**です。重要なのは、このスクリプトを**ユーザーが書くのではなく、タスク記述から Claude が自動生成する**点です。生成されたスクリプトは会話とは隔離された専用ランタイムで実行され、その間もセッションは応答可能なまま保たれます。

ここで決定的なのが**中間結果の置き場所**です。通常のエージェント実行では、各ステップの出力がすべて Claude のコンテキストウィンドウに積み上がり、長大なタスクではコンテキストが枯渇します。Dynamic Workflows では、**中間結果はスクリプトの変数（script variables）に保持され、Claude のコンテキストには最終回答だけが返ります**。これにより、数百ファイルの移行のような、1つのコンテキストには到底収まらない規模のタスクが現実的になります。

### サブエージェント・スキルとの違い ＝「プランをどこに置くか」

Claude Code には既にサブエージェントやスキルがありますが、Dynamic Workflows との違いは一点に集約されます——**「プラン（ループ・分岐・中間結果の管理）を誰が持つか」**です。

| 仕組み | プランの所在 | 向くタスク |
|:---|:---|:---|
| **通常の対話／サブエージェント** | Claude のコンテキスト（モデルが逐次判断） | 探索的・対話的で、流れが事前に読めない作業 |
| **スキル** | スキル定義（手順は固定、判断はモデル） | 手順が定型化できる繰り返し作業 |
| **Dynamic Workflows** | **コード（スクリプト側のループ・分岐）** | 構造が決まった大規模ファンアウト（移行・監査・多視点検証） |

つまり Dynamic Workflows は、**「制御フローを model-driven から code-driven に移す」**ための仕組みです。ループ・分岐・ファンアウトが決定論的にコードで回るため、1000単位のサブエージェントを安定して捌けます。

## 3. 3つの起動方法

Dynamic Workflows の起動には3つのトリガーがあります。

| 起動方法 | 動作 | スコープ |
|:---|:---|:---|
| **`ultracode` キーワード** | プロンプト中に `ultracode` を含めると、**そのタスク1つだけ**をワークフロー化。Claude Code は入力中に該当単語を**紫色（violet）にハイライト**する | 単発タスク（セッションの effort は変えない） |
| **`/effort ultracode`** | セッション中の実質的タスクごとに、Claude が**自動でワークフローを計画** | セッション全体（新セッションでリセット） |
| **保存済みワークフローコマンド** | バンドル済みの **`/deep-research`** や、自分で保存したワークフローを実行 | コマンド単位 |

`ultracode` キーワードの誤爆（ワークフロー化したくないのに単語が入ってしまった場合）は **`alt+w`**（またはトリガー語入力直後の **backspace**）で無視でき、`/config` の「Workflow keyword trigger」でトリガー自体をオフにもできます。

> なお、このトリガー語は当初 `workflow` でしたが、**v2.1.160 で `ultracode` に改称**されました。`/config` のトグル名が「Workflow keyword trigger」のままなのは、機能名（ワークフロー）に由来する名残です。

バンドルされた **`/deep-research`** は Dynamic Workflows の好例です。複数角度からWeb検索をファンアウト → ソースを取得 → 相互照合（cross-check）→ 各主張に投票 → 照合に残らなかった主張を除外し、**引用付きレポート**を返す、という多段パイプラインが1コマンドにまとまっています。

## 4. 実行の仕組み（アーキテクチャと制約）

### ランタイムの隔離とリソース上限

Dynamic Workflows のランタイムは、会話とは**隔離された環境**でスクリプトを実行します。並列度には明確な上限があります。

| 制約 | 値 | 理由 |
|:---|:---|:---|
| **同時実行エージェント数** | 最大 **16**（CPUコアが少ないと減少） | ローカルリソース使用量の制限 |
| **総計エージェント数 / 1 run** | 最大 **1,000** | 暴走ループ（runaway loops）の防止 |
| **規模感の目安** | 1 run あたり数十〜数百エージェント | — |

つまり「最大1000」は**1回の実行を通じた延べ数**であり、**同時に1000体が走るわけではありません**。実効的な同時数はマシンのCPUコア数に依存します（公称上限16、コア数が少ないマシンではさらに少なくなります）。

### v2.1.172 — サブエージェントの多階層生成（ツリー型実行へ）

当初の実行モデルは、生成スクリプト（オーケストレータ）が多数のサブエージェントを**フラットにファンアウト**する形が基本でした。**v2.1.172（2026-06-10 PT）** で、**サブエージェント自身が自分のサブエージェントを生成できる**ようになり、**最大5階層**まで入れ子にできるよう拡張されました（公式changelog: *"Sub-agents can now spawn their own sub-agents (up to 5 levels deep)"*）。

これにより、フラットなファンアウトに加えて、**親 → 子 → 孫…とタスクを段階的に分割していくツリー型（階層型）の実行**が表現できます。たとえば「大分類ごとにサブエージェントを立て、各サブエージェントがさらに細目ごとに子エージェントへ分割する」といった構造を組めます。

ただし注意点として、**同時最大16・総計最大1000 / run というリソース上限は引き続き全体に適用**されます。階層を深くしても**走る延べエージェント数の上限（1000）は変わらない**ため、各階層で子を増やしすぎると総計上限に早く到達します。深いツリーを組む場合は、階層×分岐数の掛け算で延べ数が膨らむ点を見込んでおく必要があります。

### スクリプトは「調整役」、実作業はエージェントが行う

ワークフロースクリプト自身は、**ファイルシステムやシェルに直接アクセスしません**。実際のファイル読み書き・コマンド実行・Web取得は**各サブエージェントが担い、スクリプトは調整役（オーケストレータ）に徹します**。この分離が、決定論的な制御フローとエージェントの柔軟性を両立させています。

### 権限モデル — acceptEdits と allowlist 継承

長時間・多並列で動くため、権限の扱いは把握しておくべきです。

- ワークフローが生成するサブエージェントは**常に `acceptEdits` モード**で動作し、ユーザーのツール許可リスト（allowlist）を継承します。
- **ファイル編集は自動承認**されます。
- 許可リスト外のシェルコマンド・Web fetch・MCPツールは、実行中にプロンプトを出し得ます（`claude -p` / Agent SDK ではプロンプトなしで実行）。

→ 長時間 run を無人で回す場合は、**必要なコマンドを事前に許可リストへ追加**しておくのが実務上のコツです。

### v2.1.178 — サブエージェントをモデル単位で制御（`Agent(model:opus)` 構文）

**v2.1.178（2026-06-15）** で、権限ルールに **`Tool(param:value)` 構文**（`*` ワイルドカード対応）が追加されました（公式changelog: *"Added `Tool(param:value)` syntax for permission rules to match a tool's input parameters ... e.g. `Agent(model:opus)` to block Opus subagents"*）。これにより、例えば **`Agent(model:opus)` でワークフローが生成する Opus サブエージェントの起動をブロック**するなど、**サブエージェントのモデルを権限ルールで制御**できます。大量並列でコストが跳ね上がりやすい Dynamic Workflows において、「高価なモデルのサブエージェントを禁止し、機械的処理は安価なモデルに寄せる」というコスト・暴走対策を、パーミッション側からも効かせられる実務的な追加です。

あわせて v2.1.178 では **auto モードでサブエージェントの起動が「起動前」に分類器（classifier）で評価**されるようになり（*"subagent spawns are now evaluated by the classifier before launch"*）、サブエージェントがレビューを経ずにブロック対象のアクションを要求できてしまう抜け穴が塞がれました。多階層・多並列で動く Dynamic Workflows の安全性が一段強化されています。

### 再開（resume）は同一セッション内のみ

各エージェントの結果が追跡されているため、**同一セッション内であればワークフローは再開可能**です。完了済みエージェントはキャッシュ結果を返し、残りだけがライブ実行されます。ただし **Claude Code を終了すると、次回セッションではワークフローは最初からやり直し**になる点に注意してください。

### ランタイムの制約まとめ

| 制約 | 内容 |
|:---|:---|
| **実行中のユーザー入力** | 不可。段階間で承認が要るなら、各段階を別ワークフローに分割する |
| **スクリプトからの直接I/O** | 不可（FS/シェルアクセスはエージェント経由） |
| **同時実行** | 最大16（マシン依存で減少） |
| **総計** | 最大1,000 / run |
| **階層（ネスト）** | 最大5階層（v2.1.172〜。サブエージェントが子エージェントを生成可） |
| **再開** | 同一セッション内のみ |

## 5. `/workflows` で監視・制御する

実行中・完了済みのワークフローは **`/workflows`** コマンドで一覧表示し、選択して進捗ビューを開けます。進捗ビューは**フェーズごとに「エージェント数」「累計トークン数」「経過時間」**を表示し、フェーズへドリルインすれば各エージェントのプロンプト・直近のツール呼び出し・結果まで確認できます。

| キー | 操作 |
|:---|:---|
| `↑` / `↓` | 選択 |
| `Enter` / `→` | ドリルイン（フェーズ・エージェント詳細へ） |
| `Esc` | 戻る |
| `j` / `k` | スクロール |
| **`p`** | 一時停止 / 再開 |
| **`x`** | エージェント停止 / ワークフロー全体停止 |
| `r` | 実行中エージェントの再起動 |
| **`s`** | スクリプトをコマンドとして保存（再利用可能化） |

入力ボックス下の **task panel** にも進捗の1行サマリーが出ます。`/workflows` からはいつでも停止でき、**停止しても完了済みの作業は失われません**。Desktop アプリでは、起動時に承認カード（ワークフロー名・フェーズ一覧・トークン使用の注意喚起、Once / Always / Deny）が表示され、Background tasks サイドペインで進捗を追えます。

特筆すべきは **`s` キーでのスクリプト保存**です。一度生成された有用なワークフローをコマンドとして保存すれば、以降は `/your-workflow` のように呼び出せます。`/deep-research` と同じ仕組みを、自分のワークフローでも作れるわけです。

## 6. ultracode と effort レベルの正確な関係（混同注意）

ここは最も誤解されやすいポイントです。`ultracode` は **effort レベルの一種ではありません**。

### Opus 4.8 の effort レベルは5段階

| レベル | 用途 | 永続性 |
|:---|:---|:---|
| `low` | 短く限定的、レイテンシ重視・知能非依存タスク | セッション間で永続 |
| `medium` | コスト重視で多少知能を犠牲にできる作業 | 永続 |
| **`high`** | トークンと知能のバランス（**Opus 4.8 の既定**） | 永続 |
| `xhigh` | より深い推論・高トークン消費（Opus 4.7 の既定） | 永続 |
| `max` | 最深の推論・トークン上限なし。ただし過剰思考で収穫逓減 | **セッション限定** |

### ultracode は「effort レベル」ではなく「Claude Code の設定」

公式の定義はこうです——**ultracode は `xhigh` をモデルに送りつつ、実質的タスクごとに Claude に動的ワークフローをオーケストレーションさせる設定**。

| 項目 | ultracode |
|:---|:---|
| 正体 | **Claude Code の設定**（effort レベルではない） |
| 動作 | `xhigh` 推論 ＋ タスクごとに自動でワークフローを計画 |
| 永続性 | **セッション限定**（`/effort high` で通常作業に戻す） |
| 設定方法 | `/effort ultracode`、`--settings` / Agent SDK で `"ultracode": true` |
| 注意 | 1リクエストが「理解→変更→検証」の連続ワークフローになり得る。**全タスクに適用されるため、トークン消費・所要時間が増える** |

つまり、単発のタスクだけワークフロー化したいなら **`ultracode` キーワード**（プロンプトに一語添える）、セッション全体を高難度モードで自動オーケストレーションしたいなら **`/effort ultracode`**（スラッシュコマンド）、と使い分けます。トリガー語とeffort設定で同じ `ultracode` の語を共有する点は紛らわしいので注意してください——**プロンプト本文に書けば単発トリガー、`/effort ultracode` と打てばセッション全体の設定**です。

> 補足: `effortLevel` 設定ファイルに書けるのは `low/medium/high/xhigh` のみで、`max` と `ultracode` は書けません（セッション限定のため）。また、claude.ai などチャットUI上の effort 表記（「extra」「max」等）と Claude Code/API のトークン名（`xhigh` 等）は完全一致しない場合があります。Claude Code では `xhigh`、チャットUIでは「extra」に相当、と読み替えてください。

## 7. 品質を上げる反復パターン

Dynamic Workflows の真価は「多数のエージェントを並列で走らせる」ことそのものではなく、**結果の品質を上げる反復パターンをコードで組める**ことにあります。公式が挙げる代表的パターンは次のとおりです。

- **敵対的レビュー（adversarial review）**: 独立したエージェントが互いの結果を**反証する側**に回る。多数決で疑わしい結論を棄却し、もっともらしいが誤った findings を排除する。
- **複数案の起草と比較**: 同じ課題に対し、異なる角度から複数のプランを並列起草し、相互比較して最良案へ収束させる（1案を逐次改善するより解空間を広く探れる）。
- **収束まで反復（iterative convergence）**: 答えが安定するまで探索を繰り返す。未知サイズの発見タスク（バグ掃討など）で、単純な「上位N件」より取りこぼしが減る。

これらは「数を増やす」のではなく「**多様な視点を突き合わせて確からしさを上げる**」設計であり、`/deep-research` の「投票で残らなかった主張を除外」もこの系譜です。

| **大規模移行・モダナイゼーション** | 500ファイル規模〜数千ファイルの言語・フレームワーク移行 |
| **多ソース照合リサーチ** | 複数ソースを cross-check して引用付きレポート化（`/deep-research`） |
| **コミット前の複数案検討** | 独立した複数の角度からハードなプランを起草・比較 |
| **最適化・セキュリティ監査** | プロファイラ誘導の最適化監査、セキュリティ監査 |
| **クリティカルな変更の独立検証** | 敵対的テスト・独立検証を要する作業 |

象徴的な事例として、MarkTechPost は Jarred Sumner のワークフローが **Bun のコードを Zig から Rust へ移植**し、**約11日間で約75万行を生成、既存テストスイートの99.8%をパス**したと報じています（数値は MarkTechPost 報道に基づく）。公式も「Claude Code は Opus 4.8 とともに、数十万行規模のコードベース移行をキックオフからマージまで、既存テストスイートを基準に実行できる」と謳っています。

## 9. コストと注意点

Research Preview ゆえの制限と、何より**トークンコスト**を理解しておくことが重要です。

- **トークン消費が大きい**: 公式は「1 run が、同じタスクを会話でこなすより著しく多くのトークンを消費し得る」と明記。最大1000エージェントを起動し得るため、**コストが急速に膨らみ得ます**。run は通常セッション同様にプランの使用量・レート制限にカウントされます。
- **暴走防止の設計**: 「総計1000」の上限自体が暴走ループ防止策です。加えて実行前の**プラン承認プロンプト**（フェーズ一覧表示、Yes / View raw script / No）、`/workflows` からの**一時停止・全体停止・エージェント単位の停止**が用意されています。
- **モデルコストの制御**: 各エージェントはセッションのモデルを使います。大きな run の前に `/model` を確認し、不要な段階は小型モデルにルーティングするよう依頼すると節約できます。
- **スコープを絞ったテストから**: 公式・Anthropic ブログともに「スケール前にスコープを絞ってテストせよ」と注意喚起しています。

### Opus 4.8 / Fast mode の料金（参考）

| モデル | 標準（in/out per MTok） | Fast mode |
|:---|:---|:---|
| **Opus 4.8** | $5 / $25（Opus 4.7 据え置き） | $10 / $50（標準の2倍単価・約2.5倍速） |

Fast mode は「**前モデルの Fast mode より3倍安い**」とされます（「標準の2倍単価」と「前モデル比3倍安」は別軸の話なので混同に注意。金額は The New Stack / VentureBeat 報道に基づき、最終的には公式料金ページでの確認を推奨）。

## 10. 既存機能との棲み分け

Dynamic Workflows は単体で完結する機能ではなく、Claude Code の他のオーケストレーション機能と層をなします。詳細は各専用記事に譲り、ここでは位置づけだけ整理します。

| 機能 | 役割 | Dynamic Workflows との関係 |
|:---|:---|:---|
| **Routines** | スケジュール / イベント駆動で「**いつ**動かすか」を定義（クラウド無人実行） | Routine 発火 → ワークフロー実行、という上位層の連携が可能 |
| **`/goal`** | 1セッション内で「**どこまで**動くか」を定義 | 目標達成までの自動継続。ワークフローは「どう並列に解くか」の層 |
| **`/code-review`（旧 `/ultrareview`）** | クラウドのマルチエージェント深掘りレビュー | 同じマルチエージェントでも役割が異なる（レビュー特化 vs 汎用オーケストレーション） |
| **Agent View（`claude agents`）** | 全バックグラウンドセッションの一元管理 | ワークフローのサブエージェント群もここで俯瞰できる |
| **Monitor / Push 通知** | 背景プロセスの監視・離席中の通知 | 長時間ワークフローの「全体収束」を通知するパターンに有効 |

→ Routines・Agent View・Monitor/Push の詳細は[Claude Code Monitorツール & Push通知 完全ガイド](/mdTechKnowledge/blog/claude-code-monitor-push-notification/)、Routines 自体の設定は専用記事を参照してください。

## 11. バージョン補足 — v2.1.160 で入った周辺の変更

トリガー語の改称（[§はじめに](#はじめに--ultracodeと書くだけで何が起きるのか)）と同じ **v2.1.160（2026-06-01 PT）** では、無人で長時間ワークフローを回す際に関係する安全側の変更も入りました（[公式changelog](https://code.claude.com/docs/en/changelog)）。

| 変更 | 内容 |
|:---|:---|
| **シェルstartupファイルへの書き込み確認** | `.zshenv` / `.zlogin` / `.bash_login` および `~/.config/git/` への書き込み前にプロンプトを表示。意図しないコマンド実行を防ぐための安全側の強化 |
| **`acceptEdits` のビルド設定ファイル確認** | `acceptEdits` モードでも、コード実行を許す可能性のあるビルドツール設定ファイル（`.npmrc` / `.yarnrc*` / `bunfig.toml` / `.bazelrc` / `.pre-commit-config.yaml` / `.devcontainer/` 等）への書き込み前にプロンプトを表示 |
| **Edit の read-before-edit 緩和** | 単一ファイルに対する `grep` / `egrep` / `fgrep` で内容を確認済みなら、別途 `Read` しなくても Edit が通る（read-before-edit チェックを満たす） |

> 補足: ワークフローのサブエージェントは `acceptEdits` で動作するため、上の `acceptEdits` ビルド設定ファイル確認は、無人 run 中に当該ファイルへ書き込もうとするとプロンプトが出得ることを意味します。長時間 run を無人で回す場合は、必要なら事前に許可リストへ追加しておくと中断を避けられます。

## まとめ

- **Dynamic Workflows ＝ サブエージェントを束ねる自動生成 JavaScript スクリプト**。制御フローを model-driven から **code-driven** に移すことで、1つのコンテキストに収まらない大規模タスクを捌く。
- 起動は3通り：**`ultracode` キーワード**（単発）／**`/effort ultracode`**（セッション全体）／**保存済みコマンド**（`/deep-research` 等）。
- 並列度は **同時最大16・総計最大1000 / run**。実効同時数はCPUコア依存。中間結果はスクリプト変数に保持され、コンテキストには最終結果のみ返る。**v2.1.172 からはサブエージェントが子を生成でき、最大5階層のツリー型実行が可能**（総計1000の上限は据え置き）。
- **`/workflows`** でフェーズ別のエージェント数・累計トークン・経過時間を監視し、`p`/`x` で一時停止・停止、`s` でスクリプト保存。
- **`ultracode` は effort レベルではなく Claude Code の設定**（`xhigh` ＋ 自動オーケストレーション、セッション限定）。
- トークンコストが大きく**急膨張し得る**ため、スコープを絞ったテストと `/model` 確認から始めるのが鉄則。Research Preview ゆえ仕様は変わり得る。

「ultracode」と一語添えるだけで、数百のエージェントが並列に思考し、互いを反証し、収束する——Dynamic Workflows は、Claude Code を「1人のアシスタント」から「**自律的に編成されるエージェント群**」へと押し上げる機能です。
