## 1. はじめに — なぜ2リポジトリ構成なのか

源内は **`genai-web`** と **`genai-ai-api`** の2つのリポジトリに分かれています。これは単なるディレクトリ分割ではなく、**「インターフェース」と「AI アプリ本体」を疎結合に保つ設計判断** です。

| 観点 | 一体型（GenU 等） | 2リポジトリ構成（源内） |
|---|---|---|
| **AI アプリ追加** | コード修正＋再デプロイ | **GUI から URL 登録** |
| **AI アプリ責任分界** | フロントエンドチーム任せ | **AI チームが独立して開発・運用** |
| **クラウド選択** | UI と同一クラウド | **UI は AWS、AI は別クラウドも可** |
| **障害影響** | UI 障害が AI まで波及 | **AI アプリ単独で障害切り分け可** |
| **アプリストア型展開** | 困難 | **将来的に AI アプリの市場化が可能** |

「**インターフェースは標準化、AIアプリは多様化**」という方針が、2リポジトリ構成の根幹です。

```

│  └────────────────────┘    └──────────────────────────────┘ │
│                              │                              │
│   ┌──────────────────────────┘                              │

               │ REST API（ExApp プロトコル）
               ▼
┌─────────────────────────────────────────────────────────────┐
│       源内AIアプリ (genai-ai-api) — マルチクラウド             │
│                                                             │
```

## 3. 源内Web（インターフェース層）の構成

### 3-1. リポジトリ構造

`genai-web/packages/` 直下に4ディレクトリ:

| ディレクトリ | 役割 |
|---|---|
| `cdk/` | AWS CDK によるインフラ定義（IaC） |
| `web/` | React + Tailwind フロントエンド |
| `common/` | フロント・バック共通の型定義・ユーティリティ |
| `types/` | TypeScript 型 |

`mise.toml` で Node 等のランタイムバージョンを統制、`biome.json` でリンター・フォーマッター統一されています。

### 3-2. フロントエンド技術スタック

| レイヤ | 技術 |
|---|---|
| 言語 | **TypeScript 99.1%**（ほぼ全領域） |
| UI フレームワーク | **React.js** |
| スタイル | **Tailwind CSS** + デジタル庁デザインシステム |
| ビルド | Vite（CDK でホスティング） |
| アクセシビリティ | **WCAG 2.0 AA 相当** で試験実施済 |

### 3-3. バックエンド・インフラ（AWS CDK）

源内Web は AWS にホスティングされます。CDK が起動する主要 AWS リソースは以下のとおり（推定含む）:

| AWS サービス | 用途 |
|---|---|
| **CloudFront + S3** | フロントエンド配信 |
| **Cognito** | 認証（SAML フェデレーション） |
| **API Gateway + Lambda** | バックエンド REST API |
| **DynamoDB** | チーム・AIアプリ・利用ログのメタデータ |
| **S3**（別バケット） | ファイルアップロード・生成画像 |
| **Bedrock** | 汎用アプリの LLM 呼び出し |
| **CloudWatch** | ログ・監視 |

CDK スタックは `packages/cdk/env-parameters/` 配下のパラメーターファイルでカスタマイズ可能です。

### 3-4. 認証 — SAML / Cognito の使い分け

| 認証経路 | 用途 | 実装 |
|---|---|---|
| **SAML フェデレーション** | 政府共通認証基盤 / 各組織 IdP（Okta, Entra ID 等） | Cognito User Pool の SAML IdP 連携 |
| **Cognito ローカル User Pool** | 開発・検証環境 | Cognito 標準 |

本番運用は **SAML 一本** に寄せる前提で、組織のシングルサインオン（SSO）から流入させる設計です。

## 4. 源内AIアプリ（マイクロサービス層）の構成

### 4-1. リポジトリ構造

`genai-ai-api/` 直下にクラウド別ディレクトリ:

| ディレクトリ | クラウド | 主言語 | IaC |
|---|---|---|---|
| `aws/` | AWS | Python | AWS CDK |
| `azure/` | Microsoft Azure | Python + TypeScript | **Bicep** |
| `google-cloud/` | Google Cloud | Python | **Terraform (HCL)** |

各クラウドのマネージド AI サービスや LLM API を直接利用するテンプレートが揃っています。

### 4-2. 公開済みアプリ（v1.0.3 時点）

| クラウド | アプリ | 内容 |
|---|---|---|
| **AWS** | `query-expansion-rag` | 行政実務用 RAG（検索拡張生成）の開発テンプレート |
| **Azure** | `genai-azure` | LLM をセルフデプロイして利用する開発テンプレート |
| **Google Cloud** | `lawsy-custom-bq` | 最新の法律条文データを参照・回答する法制度RAG |

それぞれが独立した Python アプリで、源内Web からは **ExApp プロトコル**で呼び出されます。

## 5. ExApp プロトコル — 2層をつなぐ仕様

### 5-1. ExApp とは

ExApp（External Application）は、源内Web と源内AIアプリをつなぐ **REST API ベースの呼び出し規約** です。

| 観点 | 内容 |
|---|---|
| **登録方法** | 源内Web の「チーム管理」画面から URL とリクエスト形式 JSON を登録 |
| **通信** | HTTPS REST |
| **認証** | チーム単位で Web ↔ AIアプリ間に共有シークレット |
| **対応** | 同期処理 / 非同期処理 |
| **入力 UI 自動生成** | リクエスト形式 JSON から動的にフォーム生成 |

### 5-2. 入力フィールド型

リクエスト形式 JSON に書く `type` は以下の8種類:

| type | 用途 |
|---|---|
| `text` | テキストフィールド（min_length / max_length 等指定） |
| `number` | 数値フィールド |
| `textarea` | 長文入力 |
| `file` | ファイルアップロード |
| `select` | セレクトボックス |
| `checkbox` | チェックボックス |
| `radio` | ラジオボタン |
| `hidden` | 非表示パラメータ |

これにより **AIアプリ側で「ユーザーが何を入力すべきか」を JSON で宣言する** だけで、源内Web が自動で UI を構築します。

### 5-3. レスポンス仕様

- **同期処理**: 即時に結果を返す（短時間処理向け）
- **非同期処理**: ジョブ ID を返し、ポーリング or Webhook で結果取得（長時間処理向け）

> 注: ExApp 仕様は 2026年3月時点のものとリポジトリで明記されており、**業界標準調査・AI開発事業者との協議で変わる可能性**がデジタル庁から告知されています。

## 6. チーム管理と権限設計

源内Web は **多階層チーム管理** を持ち、組織横断で利用統制を効かせます。

| ロール | 権限 |
|---|---|
| **システム管理者** | 全テナント横断、全 AI アプリ管理、利用ログ参照 |
| **チーム管理者** | 自チームの AI アプリ登録・メンバー追加・権限付与 |
| **チームメンバー** | 自チームに所属する AI アプリの利用 |
| **共通アプリチーム** | 全ユーザー向けの汎用 AI アプリ提供（チャット等） |

「共通アプリチーム」は特殊なチームで、源内Web に標準で付属する **汎用5アプリ**（チャット／文章生成／翻訳／画像生成／ダイアグラム生成）を全ユーザーへ提供します。

## 7. データフロー

### 7-1. 標準的なリクエストの流れ

```
1. 利用者 → 源内Web (CloudFront)
2. SAML SSO → Cognito 認可
3. Web フロントエンド → API Gateway
4. Lambda がチーム/AIアプリのメタデータを DynamoDB から取得
5. AI アプリ種別による分岐:
   ・汎用アプリ → Bedrock 直接呼び出し
   ・行政実務用 → 該当 ExApp の REST API へ転送
6. AI アプリ（マルチクラウド可）が応答生成
7. 結果を源内Web に返却 → 利用者表示
8. 全ステップを CloudWatch / 監査ログに記録
```

### 7-2. ログ・監査

| ログ種別 | 内容 |
|---|---|
| 利用ログ | ユーザー × チーム × AIアプリ × 利用時刻 |
| 入力監査 | プロンプト本文の保存（ガバナンス用） |
| エラーログ | Lambda・ExApp 側のエラー |
| メトリクス | 利用回数・トークン数・応答時間 |

監査要件が厳しい行政組織での運用を見据え、**プロンプト本文も保存**する設計が標準です（無効化も可能と推察される）。

## 8. GenU からの主要変更点

源内は AWS GenU をベースとしつつ、**「フォーク」ではなく独立した開発**として管理されています。主要差分:

| カテゴリ | GenU | 源内 | 差分の意味 |
|---|---|---|---|
| **チーム管理** | 単一テナント想定 | 多階層チーム＋ロール | 府省庁の階層構造に対応 |
| **AI アプリ管理** | コード hard-code | GUI 登録（ExApp） | アプリストア型運用 |
| **デザイン** | AWS 標準 | デジタル庁デザインシステム | 国民・公的サービスの一貫体験 |
| **アクセシビリティ** | 一般水準 | WCAG 2.0 AA 相当試験済 | 法令準拠（バリアフリー法等） |
| **多言語** | 英語中心 | 日本語・英語 | 政府職員に必須 |
| **マルチクラウド** | AWS のみ | AWS / Azure / Google | クラウド選定の自由度確保 |
| **運用機能** | 基本のみ | 監視・モニタリング・監査ログ強化 | 中央省庁の運用要件 |
| **コードベース** | 元の構造維持 | **大幅な書き換え** | 独立した開発として継続 |

GenU の更新を取り込む構造ではないため、**源内独自の進化**が今後も続きます。

## 9. 拡張性 — 自治体や民間で参考にする場合

### 9-1. 自治体での再利用

- ExApp プロトコルは独立しているため、**自治体が独自にAIアプリだけ追加開発**して源内Webにつなぐ運用が可能
- 既存の自治体クラウド（多くが AWS / Azure）に乗せやすい
- デザインシステムは「自治体DXデザインシステム」（一部都道府県で整備中）と組み合わせ可

### 9-2. 民間での参考

- 大企業の **内製 AI ポータル** の参考実装として有用
- ExApp プロトコル（`type` ベースのフォーム自動生成）は **Generic LLM Gateway** として使い回し可能
- マルチクラウド対応のテンプレートは **クラウドベンダーロック回避設計** の参考になる

## まとめ

| 観点 | ポイント |
|---|---|
| **2層構成** | 源内Web（インターフェース）と源内AIアプリ（マイクロサービス）の疎結合 |
| **クラウド戦略** | Web は AWS、AI は AWS / Azure / Google Cloud のマルチクラウド |
| **ExApp** | REST + JSON 宣言で UI 自動生成、アプリストア型展開を可能に |
| **認証** | SAML（本番） / Cognito（開発） |
| **GenU 差分** | チーム管理・AIアプリ管理・デザイン・アクセシビリティ・運用機能を独自実装 |
| **設計思想** | 「インターフェース標準化、AIアプリ多様化」 |

次回はこのアーキテクチャを **実際にデプロイする手順**を解説します。
