## はじめに — AIは「道具」から「パートナー」へ

現代のAI活用は、**「質問して答えを得る」段階から、AI自身が自律的にタスクを完遂する「エージェント」の段階** へと急速に進化しています。

この記事では、Anthropic Applied AIチームの Hannah 氏と Jeremy 氏によるセッション **「Prompting for Agents」** の内容を整理し、AIエージェントを最大限に活用するためのプロンプト戦略を解説します。

エージェントの台頭により、プロンプトエンジニアリングは「いかに上手な言い回しを使うか」から、**「エージェントにどのような行動原理と判断基準を持たせるか」** へと焦点が移っています。本記事はその実装指針として読んでください。

---

## 1. AIエージェントとは何か — Anthropic の定義

Anthropic はエージェントを次のように定義しています。

> **「ループ内でツールを使用するモデル（model using tools in a loop）」**

つまり、単発のQ&Aではなく、**自律的にツールを呼び出し、結果を見て次の行動を決める** という反復構造を持つAIのことです。

### エージェントの2つの本質

| 特性 | 内容 |
|:---|:---|
| **自律的な動作** | タスクを与えられると、どのツールを、いつ、どのように使うべきかをAI自身が判断する |
| **継続的な改善** | ツールから得られた結果を踏まえて意思決定を更新し、タスク完了まで独立して作業を続ける |

この「ループ＋自律的判断」が、従来のチャット型AIと決定的に異なる点です。

---

## 2. エージェントを導入すべきか? — 判断のチェックリスト

**すべてのタスクにエージェントが必要なわけではありません**。むしろ、不適切な導入はリソースの無駄を招きます。Anthropic は次の3軸で判断することを推奨しています。

| 判断軸 | エージェントが向くケース | 向かないケース |
|:---|:---|:---|
| **複雑性と不確実性** | ゴールは決まっているが到達プロセスが不明確（複雑なコーディング、リサーチ等） | 手順が明確に決まっている定型ワークフロー |
| **価値の高さ** | エンジニアの時間を大幅に節約できる、レバレッジの効く高価値タスク | 単純な処理で人手のほうが速いタスク |
| **エラーの許容度** | エラーが起きても回復可能、または発見が容易（検索、コンピュータ操作等） | エラーが致命傷になる、不可逆な業務処理 |

ポイントは **「決まりきった手順は通常のワークフローで処理する／不確実性が高く、エンジニアの判断を要するタスクをエージェントに任せる」** という切り分けです。

---

## 3. エージェント・プロンプティングの4鉄則

セッションの本丸となるのが、この **4つの鉄則** です。1つずつ詳しく見ていきます。

### 鉄則① エージェントの立場で考える（Think like your agents）

最も重要な原則です。

エージェントは、開発者が用意した **「環境（ツールとその説明）」** の中だけで世界を理解します。したがって設計者は次を自問する必要があります。

> **「自分がこのエージェントの立場だったら、与えられたツールと説明だけでタスクを遂行できるか?」**

具体的なチェックポイント:

- ツール名から目的が直感的に分かるか
- 引数の説明が、初見の人にも理解できる粒度になっているか
- どのタイミングでどのツールを使うべきか、説明から推測できるか
- 人間が読んで混乱するようなツール説明は、**AIも同じように混乱する**

ツール説明は「内部仕様の覚書」ではなく、**「AIへの仕事のブリーフィング」** として書くのが正解です。

### 鉄則② 適切な「ヒューリスティック（指針）」を与える

プロンプトエンジニアリングの本質は、**言葉の選択ではなく「AIにどんな概念と行動原理を持たせるか」を設計すること** です。

特に重要なヒューリスティックの例:

| 指針カテゴリ | 内容 | プロンプトでの伝え方の例 |
|:---|:---|:---|
| **不可逆性の概念** | ユーザーに害を及ぼす取り返しのつかない操作を避ける | 「破壊的な操作（rm、DROP TABLE等）の前は必ず確認を求めること」 |
| **予算（Budget）** | 何回までツールを使ってよいか上限を持つ | 「シンプルな質問には5回以内のツール実行で答えること」 |
| **終了条件** | いつタスクを終えるか明示する | 「答えが見つかったら検索ループを止めること」 |
| **優先順位** | 矛盾する要求があったときの優先 | 「速度より正確さ、正確さより安全を優先」 |

人間が業務で **無意識にやっている判断** を、ことごとく言語化してエージェントに渡してあげるイメージです。

### 鉄則③ 思考プロセスをガイドする

Claude 4 系などの最新モデルは、ツール実行の合間に思考を挟む **「インターリーブド・シンキング（Interleaved Thinking）」** が可能になっています。これを活かす設計が重要です。

| 段階 | ガイドの内容 |
|:---|:---|
| **冒頭の思考** | 検索プロセスの計画立案、成功基準の定義、必要なステップの洗い出し |
| **ツール実行間の思考** | 得られた情報の質を評価、次に取るべき行動の決定 |
| **批判的省察** | Webの検索結果は必ずしも正しくない前提で、情報源の信頼性を吟味する |
| **終盤の思考** | 当初の成功基準に照らした最終判断、未解決事項の明示 |

「考える前にツールを叩く」のではなく、**「計画→実行→評価→次の判断」** というサイクルを言語化してプロンプトに組み込むと、精度が飛躍的に向上します。

### 鉄則④ コンテキスト窓の管理

エージェントが長時間タスクを実行すると、**メモリ（コンテキスト窓）の容量** が課題になります。

| 手法 | 内容 |
|:---|:---|
| **圧縮（Compaction）** | コンテキスト上限に達しそうになった段階で、これまでの経緯を要約し、新しいインスタンスに引き継ぐ |
| **サブエージェントの活用** | リーダーエージェントが指示を出し、サブエージェントが実行・結果要約を担当（マルチエージェント方式） |
| **構造化された記録** | 重要な決定や中間結果は別ファイル／メモリに退避し、必要時に呼び出す |
| **不要情報のドロップ** | タスク完了済みの中間ログは早めに削除、最新かつ要点のみ残す |

長期タスクで失敗するエージェントの多くは、**「能力不足」ではなく「メモリ管理の失敗」** に起因します。設計時にコンテキスト戦略を決めておくことが必須です。

## 4. 評価（Evals）の重要性 — 良いエージェントは"測れる"エージェント

エージェント開発において、評価（Evals）は不可欠です。ただし分類タスクのように **単純な正解／不正解では測れない** ことが多く、独自の工夫が要ります。

### 4-1. スモールスタートの原則

最初から何百もの自動テストケースを作る必要はありません。

| 段階 | やること |
|:---|:---|
| **第1段階** | 数個の現実的なテストケースを用意し、手動で動作を確認 |
| **第2段階** | 失敗パターンを洗い出し、エッジケースを少しずつ追加 |
| **第3段階** | 安定してきた段階で自動評価へ移行 |

最初から完璧な評価フレームを作ろうとすると、**評価の構築自体が目的化** して開発が止まります。まずは少数の手動評価から始めるのが現実的です。

### 4-2. LLMを「審判」にする手法

近年強力なのが、**「別のLLMをジャッジとして使う」** アプローチです。

| 構成 | 役割 |
|:---|:---|
| **エージェント本体** | タスクを実行する |
| **ジャッジLLM** | 別途与えたルーブリック（評価基準）に基づき、エージェントの出力やプロセスが適切かを評価する |

ジャッジLLMに渡すルーブリック例:

- 出力の正確性（事実誤認がないか）
- プロセスの妥当性（不要なツール呼び出しはないか）
- ヒューリスティック遵守（不可逆操作の前に確認したか）
- 終了タイミング（無駄に検索を続けていないか）

人間が全てをチェックするのは現実的でない規模になっても、**ジャッジLLM経由で大規模に評価できる** ようになります。

## 5. 実装のすすめ — Bare Bones から始める

最後に、Hannah 氏と Jeremy 氏のセッションで強調されていた **実装の心得** を紹介します。

> **最初から複雑すぎるプロンプトを作る必要はない。**
> **まずは「素（bare bones）」のプロンプトとツールで試し、失敗箇所（エッジケース）を見つけてから、徐々に指示や例を追加していくのが成功への近道。**

| ステップ | 内容 |
|:---|:---|
| **1. 最小構成で動かす** | システムプロンプト1つ、必要最小限のツールだけで試す |
| **2. 失敗を観察する** | 実際にエージェントを走らせ、どこで誤判断するか記録する |
| **3. 失敗箇所だけ補強する** | 観察した失敗にピンポイントで対応する指示や例を追加 |
| **4. 反復する** | 上記を繰り返し、プロンプトを徐々に育てる |

最初から「全パターン網羅の完璧なプロンプト」を書こうとすると、**実際には起きない問題に過剰に対応した使い物にならないプロンプト** が出来上がりがちです。**失敗ベースで育てる**ことが、結果的に最短経路になります。

## 6. 4鉄則 早見まとめ

| 鉄則 | 一言で | 設計の問い |
|:---|:---|:---|
| **① エージェントの立場で考える** | AIに見えている世界を疑似体験する | 「ツール説明だけで、自分はタスクを遂行できるか?」 |
| **② ヒューリスティックを与える** | 行動原理を言語化する | 「人間が無意識にしている判断は何か?」 |
| **③ 思考プロセスをガイドする** | 計画→実行→省察のサイクルを設計する | 「最初の思考ブロックで何を決めさせるか?」 |
| **④ コンテキスト窓を管理する** | メモリ戦略を最初に決める | 「上限に近づいたとき、何を残し何を捨てるか?」 |

---

## まとめ — Claudeを"Claudeらしく"使う

AIエージェントの時代におけるプロンプト設計は、もはや言葉のテクニックではなく、**「自律的な存在に、どのような世界観・判断基準・思考習慣を持たせるか」** という設計行為です。

本記事の要点を最後に圧縮すると:

- **エージェントは「ループ内でツールを使うモデル」** という前提で設計する
- **エージェントを導入するか自体を判断する**（複雑性 / 価値 / エラー許容度）
- **4鉄則** を守る: 立場で考える / ヒューリスティック / 思考ガイド / コンテキスト管理
- **評価** は最初は手動・小規模で。LLMジャッジは強力な拡張手段
- **Bare Bones から始め、失敗ベースで育てる**

「最初から複雑すぎるプロンプトを書かない」「Claudeを Claude らしく使う」 — これがAnthropic自身が推奨する最短ルートです。
