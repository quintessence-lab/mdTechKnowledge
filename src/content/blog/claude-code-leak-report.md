**作成日:** 2026年4月1日
**対象事件:** 2026年3月31日 Claude Code npmパッケージ（v2.1.88）ソースマップ流出

---

## 1. 事件概要

### 何が起きたか

2026年3月31日、Anthropicのフラッグシップ製品であるClaude Codeの完全なソースコードが、公開npmレジストリを通じて流出した。ハッキングやリバースエンジニアリングではなく、**ヒューマンエラーによるパッケージング問題**が原因。

- **流出ファイル:** 59.8MBのJavaScriptソースマップファイル（cli.js.map）
- **対象パッケージ:** `@anthropic-ai/claude-code` v2.1.88
- **規模:** 約512,000行のTypeScriptコード、約1,900ファイル
- **発見者:** Solayer LabsのインターンChaofan Shou氏（X上で公開）

### 原因

Claude CodeはBun（Anthropicが2025年末に買収）上で構築されており、Bunはデフォルトでソースマップを生成する。リリース担当者が `.npmignore` に `*.map` を追加し忘れたか、`package.json` の `files` フィールドでデバッグ用ファイルを除外しなかったことが原因。なお、Bunには本番モードでもソースマップが配信されるバグ（oven-sh/bun#28001、3月11日報告・未修正）が存在していた可能性も指摘されている。

### Anthropicの公式見解

> 「本日、Claude Codeのリリースに一部の内部ソースコードが含まれました。機密の顧客データや認証情報の関与や流出はありません。これはヒューマンエラーによるリリースパッケージングの問題であり、セキュリティ侵害ではありません。再発防止策を展開しています。」

### 流出後の対応

- npmパッケージの即座のアップデート（ソースマップ削除）
- 旧バージョンのnpmレジストリからの削除
- GitHubミラーリポジトリへのDMCA取り下げ要請
- ただし、IPFS上のテレメトリ除去・ガードレール解除版も出現し、完全な封じ込めは困難

### 背景：5日間で2度目の情報流出

3月26日にはCMS設定ミスにより、未公開の「Claude Mythos」モデル詳細、ドラフトブログ記事、3,000件の未公開アセットが流出していた。1週間で2度の情報流出は、「セーフティファースト」を標榜するAnthropicにとって深刻な信頼問題を引き起こしている。

## 2. コードベースの全体構造

### アーキテクチャ概観

外部からはシンプルなCLIツールに見えるClaude Codeは、内部的には785KBのmain.tsxエントリーポイント、カスタムReactターミナルレンダラー、40以上のツール、マルチエージェントオーケストレーションシステム、バックグラウンドメモリ統合エンジンなどを含む完全なプラットフォームである。

### 主要ディレクトリ構造

```
src/
├── main.tsx              # CLIエントリーポイント（Commander.js + Ink）
├── commands.ts           # コマンドレジストリ
├── tools.ts              # ツールレジストリ
├── QueryEngine.ts        # コアLLMエンジン（約46,000行）
├── context.ts            # コンテキスト管理
├── cost-tracker.ts       # トークン追跡
│
├── commands/             # 約50コマンド
├── tools/                # 約40ツール
├── components/           # 約140 UIコンポーネント
├── hooks/                # Reactフック
├── services/             # 外部連携
├── screens/              # フルスクリーンUI
│
├── bridge/               # IDE統合
├── coordinator/          # マルチエージェントオーケストレーション
├── plugins/              # プラグインシステム
├── skills/               # スキルシステム
├── voice/                # 音声入力
├── remote/               # リモートセッション
├── server/               # サーバーモード
├── memdir/               # 永続メモリ
├── tasks/                # タスクシステム
├── buddy/                # コンパニオンペット
├── vim/                  # Vimモード
└── native-ts/            # ネイティブTypeScript
```

### ツールシステム

40以上のツールが自己完結型で、各ツールにスキーマ・権限チェック・実行ロジックがある。

- **コア:** Shell実行（Bash/PowerShell）、ファイル読み書き・編集、glob/grep検索、Jupyter Notebook編集、インタラクティブVMシェル
- **Web:** Webフェッチ、Web検索、Playwright経由のブラウザオートメーション
- **エージェント:** 子エージェント生成、バックグラウンドタスク管理、エージェント間メッセージ、エージェントチームの作成・削除
- **インフラ:** MCPリソースアクセス、cronスケジューリング、リモートエージェントトリガー、ワークフロー実行、スリープ・自己再開
- **内部専用:** ConfigTool、TungstenTool（Anthropic社員限定）
- **KAIROS専用:** ファイル配信、プッシュ通知、PR監視

### コンテキスト圧縮戦略（3層圧縮）

1. **MicroCompact:** APIコール不要。キャッシュされたコンテンツをローカルで直接編集し、古いツール出力を除去
2. **AutoCompact:** コンテキストウィンドウ上限に近づくとトリガー。13,000トークンのバッファを確保し、最大20,000トークンの要約を生成。3回連続失敗後のサーキットブレーカー搭載
3. **3層メモリアーキテクチャ:**
   - インデックス（常時ロード、1行約150文字のポインタ）
   - トピックファイル（必要時にフェッチ）
   - セッショントランスクリプト（検索可能だが直接ロードされない）

## 4. 主要未公開機能の詳細

### 4-1. KAIROS（カイロス）

ギリシャ語の「正しい瞬間（καιρός）」に由来。ソース内で**150回以上参照**される最大の未公開機能。

**基本動作:**
- 24時間365日バックグラウンドで動作する永続的なデーモンモード
- 数秒ごとに `<tick>` ハートビートプロンプトを受信し「今やるべきことがあるか？」を判断
- ユーザーの操作なしに、エラー修正、メッセージ応答、ファイル更新、タスク実行が可能
- **15秒ブロッキング予算** — ユーザーを15秒以上中断させるアクションは延期

**KAIROS専用ツール（通常のClaude Codeにはない）:**
1. **プッシュ通知:** ターミナルが閉じていてもデスクトップ/スマホに通知
2. **ファイル配信:** 要求なしにファイルを直接ユーザーに送信
3. **PRサブスクリプション:** GitHubウェブフックを監視し、PRの変更に自動対応

**ロギング:**
- 追記専用（append-only）のデイリーログファイルを維持
- 観察、判断、アクションを記録
- 自身の履歴を消去することは不可能

**信頼モデル:**
- KAIROSはプロアクティブなループを導入し、エージェント自身が「何をすべきか」を判断する必要がある
- これは根本的に異なる信頼モデルを要求する

### 4-2. autoDream（夢モード）

KAIROS内のサブシステムとして動作するバックグラウンドメモリ統合エンジン。

**トリガー条件（3ゲートシステム — 3つすべて通過が必要）:**
- ゲートの具体的条件はソース内に定義

**4フェーズの動作:**
1. **Orient（方向付け）:** メモリファイルの読み込み
2. **Gather Recent Signal（最新情報収集）:** 保持すべき新情報の発見
3. **Consolidate（統合）:** メモリファイルの書き込み・更新、相対日付の絶対日付への変換、矛盾する事実の削除
4. **Prune and Index（剪定・索引）:** メモリを200行・25KB以下に維持、矛盾の解消

**内部プロンプト:**
> 「あなたは夢を実行しています。メモリファイルに対する反省的なパスです。最近学んだことを永続的で整理されたメモリに統合し、将来のセッションが素早く方向付けできるようにしてください。」

**安全設計:**
- フォークされたサブエージェントとして実行（メインエージェントのthought streamを汚染しない）
- 読み取り専用アクセス（プロジェクトを閲覧可能だが変更は不可）
- メモリは「ヒント」として扱われ、使用前に検証される

### 4-3. ULTRAPLAN（ウルトラプラン）

複雑な計画タスクを外部にオフロードする高度な計画モード。

- リモートのCloud Container Runtime（CCR）セッションでOpus 4.6を実行
- 最大**30分間**の専用思考時間を付与
- ブラウザUIからリアルタイムで計画プロセスを監視可能
- ターミナルは3秒ごとにポーリング
- 承認後、特殊なセンチネル値 `__ULTRAPLAN_TELEPORT_LOCAL__` で結果をローカルターミナルに「テレポート」
- スマートフォンやブラウザからの承認にも対応

### 4-4. BUDDY（バディ）

たまごっち風のターミナルコンパニオンペットシステム。

**ペットシステム:**
- ユーザーIDからMulberry32 PRNGで決定論的に生成（同じユーザーは常に同じバディ）
- **18種** × **5レアリティ段階:**
  - Common（60%）: Pebblecrab, Dustbunny, Mossfrog, Twigling, Dewdrop, Puddlefish
  - Uncommon（25%）: Cloudferret, Gustowl, Bramblebear, Thornfox
  - Rare: （詳細不明）
  - Epic: （詳細不明）
  - Legendary（1%）: Nebulynxなど
- **1%のシャイニー確率**（レアリティとは独立） — Shiny Legendary Nebulynxの出現確率は0.01%

**ステータスシステム:**
- 5つのステータス: DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK
- 6種類の目のスタイル
- 8種類の帽子オプション（一部はレアリティで制限）
- ASCIIアートスプライト（複数アニメーションフレーム）

**実装詳細:**
- 種名は `String.fromCharCode()` でエンコード（ビルドシステムのgrep検査を回避）
- `buddy/companion.ts` に実装
- 独自のシステムプロンプトを持ち、ユーザーの入力ボックスの横に表示
- 名前で呼びかけると応答可能
- **リリース計画:** 4月1日〜7日がティーザー期間、5月に正式ローンチ予定

### 4-5. TEAMMEM（チームメモリ）

共有チームメモリシステム。フィーチャーフラグリストに `TEAMMEM — Shared team memory` として記載。

- エージェント間で共有されるメモリ空間
- Coordinator Modeと連携し、複数のワーカーエージェント間でコンテキストを共有

### 4-6. COORDINATOR MODE（コーディネーターモード）

マルチエージェントオーケストレーションシステム。

- 1つのメインエージェントが複数のワーカーエージェントにタスクを割り当て
- ワーカーは並列で実行し、結果を報告
- **メールボックスシステム:** ワーカーが危険な操作を実行する際、リーダーに権限を要求
- **アトミッククレーム機構:** `createResolveOnce` で複数ワーカーが同一の権限リクエストを処理するのを防止
- KVキャッシュを使用したfork-joinモデルにより、並列処理がほぼ無コスト

### 4-7. その他の未公開機能

- **VOICE_MODE:** プッシュトゥトーク音声インターフェース（完全実装済み・未出荷）
- **BRIDGE_MODE:** VS Code / JetBrains IDEとの双方向通信レイヤー
- **CHICAGO_MCP:** Computer Use（内部コードネーム「Chicago」）。スクリーンショットキャプチャ、クリック/キーボード入力
- **AGENT_TRIGGERS:** cronベースのスケジュールドエージェント
- **EXTRACT_MEMORIES:** バックグラウンドメモリ抽出エージェント

## 5. 内部モデル情報

### コードネーム体系

| コードネーム | 対応モデル | 備考 |

| **Capybara** | Claude 4.6 | v8まで反復、fast/通常バリアント |

### 未公開モデルバージョン

ソースコード内に `opus-4-7`、`sonnet-4-8` への参照が存在。

### 内部ベンチマーク

- Capybara v8: **29-30%の誤主張率**（v4の16.7%から悪化 — 回帰）
- Capybara v2-fastが1Mコンテキストウィンドウ付きで準備中

### 既知の本番問題

- Capybaraがツール結果後のターン境界に似たプロンプト形状で生成を早期停止するバグ
- 対策: プロンプト形状の外科的修正、安全な境界マーカー(`Tool loaded.`)の強制挿入
- autocompact失敗による**1日あたり250,000回の無駄なAPIコール**（バグ修正コメントで発覚）

## 6. セキュリティ・防御メカニズム

### Anti-Distillation（蒸留防止）

**メカニズム1: フェイクツール注入**
- フラグ `ANTI_DISTILLATION_CC` 有効時、APIリクエストに `anti_distillation: ['fake_tools']` を送信
- サーバーがシステムプロンプトにデコイツール定義を注入
- 競合がAPI通信を記録してモデルを蒸留する場合、トレーニングデータが汚染される

**メカニズム2: サーバーサイドテキスト要約**
- ツールコール間のアシスタントテキストをバッファリング・要約
- 暗号署名付きで要約を返し、後続ターンで元テキストを復元可能

### Undercover Mode（潜入モード）

- Anthropic社員が公開/OSSリポジトリで作業する際に自動有効化
- AIであることへの一切の言及を禁止
- `Co-Authored-By` 帰属を除去
- 内部モデルコードネームの漏洩を防止

### その他のセキュリティ機能

- **ネイティブクライアント認証:** バイナリの真正性を暗号的に証明
- **権限システム:** ツールアクションをリスクレベル（Low, Medium, High）で分類。ML（機械学習）ベースの自動承認機能。6層の権限システム
- **フラストレーション検出:** 正規表現ベースの感情分析

## 8. 戦略的影響

### Anthropicへの影響

- **知的財産の流出:** 内部APIロジック、OAuth 2.0認証フロー、権限強制、マルチエージェント連携、未公開機能パイプラインが公開
- **ロードマップの露出:** コードはリファクタリング可能だが、戦略的サプライズは取り消せない
- **信頼性の毀損:** エンタープライズ収益の80%を占める中、セーフティファーストブランドが1週間で2度の流出により棄損

### 競合への影響

- エージェントハーネス設計の「教科書」が公開された
- 6ヶ月の製品サプライズが2ヶ月のレースに変化

### コミュニティの反応

- クリーンルームリライト版がGitHub史上最速（2時間で50,000スター）を記録
- Python版・Rust版への書き換えプロジェクトが始動
- 開発者の論調が「Anthropicへの批判」から「技術的称賛」に急速に転換

## 9. ユーザーへの推奨事項

1. **即座にv2.1.88より新しいバージョンにアップデート**
2. ネイティブインストーラーの使用を推奨
3. 3月31日 00:21〜03:29 UTC の間にnpmでアップデートした場合、同時期に発生したaxiosサプライチェーン攻撃（RAT注入）の影響を確認
4. 機密プロジェクトではAIモデルのローカル実行を検討
