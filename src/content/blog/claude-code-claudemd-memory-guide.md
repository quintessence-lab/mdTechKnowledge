# Claude Code の CLAUDE.md とメモリの仕組み

> Claude Code v2.1.x / 2026年4月時点

---

## 1. CLAUDE.md とは

CLAUDE.md は、Claude Code が**セッション起動時に自動で読み込む指示ファイル**。プロジェクトのルール、コーディング規約、作業手順などを記述しておくと、毎回指示し直す必要がなくなる。

### 特徴

| 項目 | 内容 |
|---|---|
| ファイル名 | `CLAUDE.md`（大文字固定） |
| 読み込みタイミング | セッション起動時に自動 |
| 書き込み | ユーザーが手動で編集、または Claude Code に依頼 |
| 形式 | Markdown（フロントマターなし） |

---

## 2. CLAUDE.md の階層構造

CLAUDE.md は**3つの階層**に配置でき、それぞれ読み込まれる条件が異なる。

```
~/.claude/CLAUDE.md                    ← ① グローバル（常に読まれる）
C:\Users\<user>\projectA\CLAUDE.md     ← ② プロジェクトA（projectAで起動した場合のみ）
C:\Users\<user>\projectB\CLAUDE.md     ← ③ プロジェクトB（projectBで起動した場合のみ）
```

### 読み込みルール

| 階層 | 配置場所 | いつ読まれるか | 用途 |
|---|---|---|---|
| **グローバル** | `~/.claude/CLAUDE.md` | **常に読まれる**（どのフォルダで起動しても） | 言語設定、全プロジェクト共通のルール |
| **プロジェクト** | `作業フォルダ/CLAUDE.md` | **そのフォルダで起動した場合のみ** | プロジェクト固有のルール、push先制限など |

### 重要なポイント

- **グローバルは全セッションに影響する** — 言語設定やデータプライバシーなど、絶対に守るべきルールを記載する
- **プロジェクトCLAUDE.mdは他のプロジェクトには見えない** — projectAのCLAUDE.mdはprojectBから起動したセッションには読まれない
- **サブエージェントも同じルールに従う** — メインエージェントが起動したサブエージェントも、同じCLAUDE.mdを読み込む

### よくある落とし穴

```
セッション起動フォルダ: projectA
操作対象: projectB のファイル

→ 読まれるCLAUDE.md: ① グローバル + ② projectA
→ projectB の CLAUDE.md は読まれない！
```

別のプロジェクトのファイルを操作する場合、そのプロジェクトのCLAUDE.mdは読まれない。全プロジェクトで守るべきルールは**グローバルに書く**必要がある。

---

## 3. CLAUDE.md に書くべき内容

### グローバル（`~/.claude/CLAUDE.md`）

- 言語設定（「すべての返答を日本語で行う」等）
- データプライバシーのルール
- コーディング原則（最小変更、シンプルさ維持など）
- 複数プロジェクトにまたがるルール（push先制限など）

### プロジェクト（`作業フォルダ/CLAUDE.md`）

- 技術スタック（使用フレームワーク、ビルドツール）
- テスト実行コマンド
- ディレクトリ構成の説明
- デプロイ手順
- 触ってはいけないファイルの指定

### 書かないほうがよい内容

- 頻繁に変わる情報（バージョン番号、担当者名など）→ メモリに保存
- コードから読み取れる情報（関数の引数の型、既存のディレクトリ構成など）

## 4. メモリの仕組み

Claude Code には**ファイルベースの永続メモリ**がある。会話をまたいで情報を記憶する仕組みで、CLAUDE.md とは別の仕組みとして機能する。

### メモリの保存先

```
~/.claude/projects/<プロジェクト識別子>/memory/
  ├── MEMORY.md          ← インデックスファイル（常に読み込まれる）
  ├── user_role.md       ← 個別のメモリファイル
  ├── feedback_xxx.md    ← フィードバック記録
  └── project_xxx.md     ← プロジェクト情報
```

`<プロジェクト識別子>` はフォルダパスから自動生成される（例: `C--Users-username-projectA`）。

### メモリの種類

| 種類 | 内容 | 例 |
|---|---|---|
| **user** | ユーザーの役割・スキル・好み | 「シニアエンジニア、React経験豊富」 |
| **feedback** | ユーザーからの修正指示・承認 | 「テスト時にモックを使わないこと」 |
| **project** | プロジェクトの状況・方針 | 「4月15日までにリリース」 |
| **reference** | 外部リソースへのポインタ | 「バグはLinearの"INGEST"で管理」 |

### メモリのライフサイクル

```
① ユーザーとの会話で情報を学習
    ↓
② メモリファイルに保存（Write ツール）
    ↓
③ MEMORY.md にインデックスを追記
    ↓
④ 次回セッション起動時に MEMORY.md を自動読み込み
    ↓
⑤ 関連がありそうなメモリファイルを必要に応じて Read
```

### メモリとCLAUDE.mdのスコープ比較

| 比較項目 | CLAUDE.md | メモリ |
|---|---|---|
| **グローバル** | `~/.claude/CLAUDE.md` → 全セッション | なし（メモリはプロジェクト単位のみ） |
| **プロジェクト単位** | `作業フォルダ/CLAUDE.md` | `~/.claude/projects/<識別子>/memory/` |
| **読み込みタイミング** | セッション起動時に全文読み込み | MEMORY.md（インデックス）は常に読み込み、個別ファイルは必要時 |
| **書き込み** | ユーザーが手動 or Claude に依頼 | Claude が自動的に判断して保存 |

### 重要：メモリはプロジェクト単位

```
~/.claude/projects/
  C--Users-username-projectA/memory/   ← projectAのメモリ
  C--Users-username-projectB/memory/   ← projectBのメモリ（別管理）
```

projectAで保存したメモリはprojectBからは見えない。プロジェクトをまたいで共有したい情報は**グローバルCLAUDE.md**に書く。

## 5. CLAUDE.md とメモリの使い分け

| 情報の性質 | 保存先 | 理由 |
|---|---|---|
| 全プロジェクト共通の原則 | グローバル CLAUDE.md | どこから起動しても読まれる |
| プロジェクト固有のルール | プロジェクト CLAUDE.md | そのプロジェクトでのみ必要 |
| ユーザーの好み・修正指示 | メモリ（feedback） | 会話から自動保存、次回以降に活用 |
| 進行中のプロジェクト状況 | メモリ（project） | 変化する情報、期限付きの情報 |
| コードから読み取れること | どちらにも書かない | コードが正本 |

### 判断フローチャート

```
その情報は全プロジェクトで必要？
  ├── はい → グローバル CLAUDE.md
  └── いいえ
        その情報はコードから読み取れる？
          ├── はい → どちらにも書かない
          └── いいえ
                その情報は固定的なルール？
                  ├── はい → プロジェクト CLAUDE.md
                  └── いいえ → メモリ
```

## 6. 実践的な設定例

### グローバル CLAUDE.md の例

```markdown
# グローバル設定

## 原則
- **言語**: すべての返答を日本語で行う
- **データプライバシー**: ローカルファイル・データを外部サービスやAPIに送信しない
- **実行テスト必須**: スクリプトを実装した際は必ず一度実行テストを行うこと

## コーディング原則
- **最小限の変更**: 必要な箇所のみを変更すること
- **シンプルさの維持**: あらゆる変更を可能な限りシンプルに保つこと

## サイト公開ルール
- **push先**: organization/repository のみ
- **他のリポジトリに記事をpushしないこと**
```

### プロジェクト CLAUDE.md の例

```markdown
# プロジェクト設定

## 技術スタック
- Astro 4.x / GitHub Pages
- ビルド: npm run build
- デプロイ: git push → GitHub Actions → 自動

## ディレクトリ構成
- src/content/blog/ — 記事（Markdown）
- public/guides/ — iframe埋め込みHTML
```
