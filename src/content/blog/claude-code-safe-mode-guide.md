## はじめに — 「設定のどれかが壊している」を一発で確かめる

Claude Code を使い込むほど、CLAUDE.md・スキル・プラグイン・hooks・MCP サーバー・カスタムコマンドと、自分専用のカスタマイズが積み上がっていきます。便利な反面、ある日「**急に挙動がおかしい。でも、どのカスタマイズが原因か分からない**」という状況に陥ることがあります。hook が想定外のタイミングで走る、MCP サーバーが固まる、特定のスキルが悪さをする——切り分けの最初の一歩は「**カスタマイズを全部止めて、素の状態でも再現するか**」を確かめることです。

これを 1 コマンドで実現するのが **Safe Mode（セーフモード）** です。Safe Mode は、ユーザー由来のカスタマイズをまとめて無効化し、**認証とモデル選択だけが効く「ベースインストール状態」**で Claude Code を起動します。Safe Mode で問題が消えれば原因はカスタマイズのどれか、消えなければカスタマイズ以外——と、原因の所在を一気に二分できます。

本記事は、Safe Mode の起動方法、無効化される対象と維持されるもの、`--bare` との違い、そしてトラブル別の切り分け手順までを体系的に解説する入門ガイドです。読み取り専用で計画を固める「[Plan Mode ガイド](/mdTechKnowledge/blog/claude-code-plan-mode-guide/)」と並ぶ「モード」シリーズの一本として読んでください。

> 本記事は2026年6月時点の Claude Code 公式ドキュメント・CHANGELOG に基づきます。Safe Mode（`--safe-mode` / `/cd` など）は **Claude Code v2.1.169 以降**で利用できます。バージョン・環境により表記や挙動が異なる場合があるため、不確実な点は「推測」と明記しています。

## 1. Safe Mode とは — 目的と位置づけ

Safe Mode は、**設定起因の不具合を切り分けるためのトラブルシューティング用起動モード**です。Plan Mode が「実行前に止まって計画を見せる」安全モードであるのに対し、Safe Mode は「**カスタマイズを読み込まずに起動する**」診断モードであり、目的がまったく異なります。

| 観点 | 通常起動 | Safe Mode |
|:---|:---|:---|
| CLAUDE.md / スキル / プラグイン | 読み込む | **読み込まない** |
| hooks / MCP サーバー | 読み込む | **読み込まない** |
| カスタムコマンド / エージェント | 読み込む | **読み込まない** |
| テーマ / キーバインド / ステータスライン | 反映される | **反映されない（既定値）** |
| 認証 / モデル選択 / 組み込みツール / 権限 | 有効 | **有効（通常どおり）** |

ポイントは、Safe Mode が「機能を制限する」のではなく「**自分が足したものだけを外す**」モードだということです。Claude Code 本体（認証・モデル・組み込みツール・権限）はそのまま動くので、「素の Claude Code は正常か？」を確かめる土台になります。

公式ドキュメントの「Debug your configuration」でも、設定不具合の調査はまず `claude --safe-mode` から始めることが推奨されています。**Safe Mode で問題が消えれば、無効化された面（surface）のどれかが原因**であり、そこから個別に絞り込んでいく、という流れです。

## 2. 起動方法 — フラグと環境変数

Safe Mode の起動方法は 2 通りあります。どちらも結果は同じで、内部的には `--safe-mode` が環境変数 `CLAUDE_CODE_SAFE_MODE` を設定します。

| 起動方法 | コマンド例 | 使いどころ |
|:---|:---|:---|
| **フラグ指定** | `claude --safe-mode` | その場の 1 セッションだけ素の状態で起動したいとき（最も手軽） |
| **環境変数** | `CLAUDE_CODE_SAFE_MODE=1 claude` | スクリプトやシェル設定で一時的に強制したいとき |

```bash
# 方法1: フラグで起動（推奨）
claude --safe-mode

# 方法2: 環境変数で起動
CLAUDE_CODE_SAFE_MODE=1 claude
```

> Windows（PowerShell）で環境変数指定する場合は `$env:CLAUDE_CODE_SAFE_MODE=1; claude` のように設定してから起動します（推測：公式は POSIX シェルの例示。原理上は同じ環境変数の設定で機能するはずですが、実環境での確認を推奨します）。

切り分けが終わったら、フラグや環境変数を付けずに `claude` を起動し直せば、通常どおりカスタマイズが読み込まれた状態に戻ります。Safe Mode は設定ファイルを書き換えるわけではないので、**起動を切り替えるだけで元に戻せる**のが利点です。

## 3. 無効化される対象 vs 維持されるもの

Safe Mode の核心は「何を外し、何を残すか」の線引きが明確に決まっていることです。公式 CLI リファレンスの記述に基づくと、次のように整理できます。

### 3-1. 無効化される対象（読み込まれない）

| カテゴリ | 無効化される対象 |
|:---|:---|
| メモリ・指示 | **CLAUDE.md**、auto-memory（自動メモリ） |
| 拡張機能 | **スキル（skills）**、**プラグイン（plugins）** |
| 自動化・連携 | **hooks**、**MCP サーバー**、LSP サーバー |
| カスタム定義 | **カスタムコマンド**、**エージェント（agents）**、ワークフロー |
| 見た目・操作 | **カスタムテーマ**、**カスタムキーバインド**、ステータスライン、出力スタイル（output styles）、ファイル候補（file-suggestion）コマンド |

要するに、**自分や組織が後から足したカスタマイズはほぼすべて外れ、ベースインストール状態になる**と考えてよいです。

### 3-2. 維持されるもの（通常どおり効く）

| 項目 | 動作 |
|:---|:---|
| **認証（authentication）** | 通常どおり（ログイン状態は維持） |
| **モデル選択（model selection）** | 通常どおり |
| 組み込みツール（built-in tools） | 通常どおり（Read / Edit / Bash など本体機能） |
| 権限（permissions） | 通常どおり |

認証とモデル選択が効くため、「ログインし直し」や「モデルを選び直す」手間なく、即座に素の状態で作業を始められます。

### 3-3. 例外 — 組織の管理設定（managed settings）

注意点として、**組織が管理ポリシー（managed settings）で配布した設定は Safe Mode でも適用され続けます**。具体的には、ポリシー設定された hooks・ステータスライン・ファイル候補コマンド、managed plugins、managed skills、managed CLAUDE.md、ポリシー設定された MCP サーバーは Safe Mode でも生きています。

| 設定の出どころ | Safe Mode での扱い |
|:---|:---|
| 自分（ユーザー）のカスタマイズ | **無効化される** |
| プロジェクトのカスタマイズ | **無効化される** |
| 組織の管理ポリシー（managed settings） | **適用され続ける** |

つまり Safe Mode は「**自分で足したもの**」を外すモードであり、組織が強制している設定までは外しません。企業環境で「Safe Mode にしたのに hook が走る」場合は、それが managed settings 由来である可能性を疑ってください。

## 4. `--safe-mode` と `--bare` の違い

Claude Code には、似て非なる「最小起動」フラグとして `--bare` もあります。両者の違いを押さえておくと、用途を取り違えずに済みます。

| 観点 | `--safe-mode` | `--bare` |
|:---|:---|:---|
| カスタマイズの無効化 | する | する |
| **認証** | **通常どおり効く** | 効かない（より素の状態） |
| **モデル選択** | **通常どおり効く** | 効かない |
| **権限・組み込みツール** | 通常どおり | より制限的 |
| 主な用途 | **設定起因の不具合切り分け** | さらに素に近い最小起動 |

`--safe-mode` は「認証・モデル選択・権限・組み込みツールは生かしたまま、カスタマイズだけ外す」という**実務的な切り分けに最適化**されたモードです。「Fable 5 から自動フォールバックが起きるのはカスタマイズのせいか？」といった、本体機能を動かしながらカスタマイズの影響を見たいケースで真価を発揮します。一方 `--bare` はより素に近い最小起動で、目的が異なります（推測：`--bare` の詳細仕様はバージョン依存。最新の正確な差分は公式 CLI リファレンスでご確認ください）。

## 5. トラブル別の切り分け手順

Safe Mode は「原因を二分する」最初の一手です。Safe Mode で再現するかどうかで、その後の調べ方が変わります。

| 症状 | まず Safe Mode で確認 | Safe Mode で消えた場合の次の一手 | Safe Mode でも再現する場合 |
|:---|:---|:---|:---|
| 起動・応答が急に不安定 | `claude --safe-mode` で再現するか | カスタマイズを 1 つずつ戻して原因特定 | 本体・認証・環境側を疑う |
| hook が予期せぬ挙動 | 再現するか確認 | `/hooks` で個別に有効化し犯人を特定 | managed settings の hook を疑う |
| MCP サーバーが固まる | 再現するか確認 | MCP 設定を 1 つずつ有効化 | ネットワーク・対象サーバー側を疑う |
| スキル/プラグインが競合 | 再現するか確認 | `/plugin` で個別に切り替えて切り分け | プラグイン以外の要因を疑う |
| Fable 5 から自動フォールバック | 再現するか確認 | カスタマイズが引き金か判定し除外 | モデル・アカウント条件を確認 |

### 切り分けの基本フロー

```
1. claude --safe-mode で起動
2. 問題が再現するか確認
   ├─ 消えた  → 原因はカスタマイズのどれか
   │            → 通常起動に戻し、疑わしい設定を1つずつ
   │              無効化/有効化して犯人を特定
   └─ 再現する → 原因はカスタマイズ以外
                  （本体・認証・ネットワーク・環境）
```

このように、Safe Mode は「**犯人はカスタマイズか、それ以外か**」を最短で二分するための診断スイッチです。やみくもに設定をいじる前に、まず Safe Mode で起動して再現性を確かめる——これが遠回りに見えて最速の切り分けです。

## 6. 同時期リリースの関連機能 — `post-session` hook と `/cd`

Safe Mode（v2.1.169）と同時期に、運用・自動化まわりの機能もいくつか追加されました。Safe Mode で hooks やセッションを扱う際の周辺知識として、2 つだけ触れておきます。

| 機能 | 概要 | Safe Mode との関係 |
|:---|:---|:---|
| **post-session hook** | セッション終了後・ワークスペース削除前に走る hook。自動化セッションで未コミット作業のスナップショットやログのエクスポートに使う（CHANGELOG 記載） | hook の一種なので Safe Mode では無効化される（ユーザー設定の場合） |
| **`/cd <path>`** | セッションを別の作業ディレクトリへ移動するコマンド。プロンプトキャッシュを保持したまま移動先の CLAUDE.md を追記する | Safe Mode 中は CLAUDE.md を読み込まないため、移動先の指示も適用されない |

- **post-session hook** は CHANGELOG に「セルフホストランナー向けに、セッション終了後・ワークスペース削除前に走る hook」として記載があります。ただし**公式の hooks リファレンスでは個別イベントとして `post-session` という名称は確認できず、ライフサイクルイベント `SessionEnd` に統合されている**ようです（推測：CHANGELOG の "post-session hook" は実体としては `SessionEnd` イベントを指す可能性が高い。正確なイベント名・設定方法は公式 hooks リファレンスで要確認）。
- **`/cd <path>`** は、セッションを新しい作業ディレクトリへ移動するコマンドです。**プロンプトキャッシュを保持**したまま、移動先ディレクトリの CLAUDE.md をシステムプロンプト再構築ではなくメッセージとして追記します。セッションは移動先のプロジェクトストレージに再配置され、`--resume` / `--continue` もそこから見つかります。初めて作業するディレクトリでは信頼確認のプロンプトが出ます。単にディレクトリへのアクセスを足したいだけなら、セッションを移動しない `/add-dir` を使います。`/cd` も **v2.1.169 以降**が必要で、それ以前のバージョンでは `Unknown command: /cd` になります。

## まとめ

| 観点 | ポイント |
|:---|:---|
| **何のモードか** | カスタマイズを全無効化し、素のインストール状態で起動する診断モード |
| **起動方法** | `claude --safe-mode` ／ 環境変数 `CLAUDE_CODE_SAFE_MODE=1` |
| **無効化される** | CLAUDE.md・スキル・プラグイン・hooks・MCP・カスタムコマンド・エージェント・テーマ・キーバインド等 |
| **維持される** | 認証・モデル選択・組み込みツール・権限（＝本体機能） |
| **例外** | 組織の managed settings は Safe Mode でも適用される |
| **使いどころ** | 設定起因の不具合切り分け、hook/プラグイン/MCP の競合特定、Fable 5 フォールバックの検証 |
| **バージョン** | Claude Code v2.1.169 以降 |

Safe Mode は、Claude Code を「**カスタマイズの塊**」から「**素の Claude にいつでも戻せる道具**」に変える機能です。設定が複雑になるほど「どれが壊しているのか分からない」事態は起こりやすくなりますが、`--safe-mode` ひとつで原因をカスタマイズ側と本体側に二分できます。挙動がおかしいと感じたら、設定をいじる前にまず Safe Mode——を習慣にしておくと、トラブル対応が一気に楽になります。
