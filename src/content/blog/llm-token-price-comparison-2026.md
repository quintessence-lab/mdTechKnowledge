## §1. なぜ今「トークンエコノミー」か — 「フロンティア一択」の終わり

2025年までの合言葉は「**とりあえず一番賢いモデルを使う**」でした。しかし2026年、状況は変わりました。

1. **モデルが階層化した**。同一ベンダー内でもフロンティア級（Opus 4.8 / Gemini 3.1 Pro / GPT-5.5）から軽量級（Haiku 4.5 / Flash-Lite / GPT-5-nano）まで、**出力単価で 10〜100 倍の開き**があります。
2. **エージェント化で消費量が桁違いに増えた**。1リクエスト＝数百トークンだった時代から、**1タスクで数百万トークン**を消費する自律エージェントの時代へ。単価差がそのまま請求額の桁差になります。
3. **コストが経営指標になった**。金融機関からは「**AI トークンコストが業績・相場の新たな変数**」との指摘も出ており、企業では **FinOps（クラウド原価管理）の対象**として AI 支出を可視化・制御する動きが加速しています。

つまり、**「賢さ」だけでなく「単価×実消費トークン数＝実効コスト」で設計する**——これがトークンエコノミーの基本です。特に企業では、**全社員が全タスクでフロンティアを叩く**運用はコスト的に持続しません。

## §3. 「Copilot」はトークン単価を持たない — 課金構造の3類型

「Copilot のトークン単価は？」という問いには**注意が必要**です。**GitHub Copilot は独自のトークン単価を持たず、上記のモデルを"再販"している**からです。Microsoft 系は次の3つで課金構造がまったく異なります。

| 製品 | 課金モデル | トークン単価 | 中身 |
|:---|:---|:---|:---|
| **GitHub Copilot** | **使用量課金**（2026-06-01〜） | **持たない（ラッパー）** | 月額＋AIクレジット枠。Pro $10（$15枠）/ Pro+ $39（$70）/ Max $100（$200）/ Business $19（$19）/ Enterprise $39。**プレミアムリクエスト＝モデル乗数**（例: **Claude Opus 級 27×**・**GPT-5 級 6×**・**code review 13×**）でクレジットを消費。コード補完・Next Edit は**無料枠**（クレジット非消費） |
| **Microsoft 365 Copilot** | **座席課金** | **なし（従量ではない）** | **Business $21/user・月**（年払い促進 $18）／**Enterprise $30/user・月**。M365 ベースライセンスへのアドオン。トークンは使い放題に近い定額 |
| **Azure OpenAI** | **トークン従量** | **あり（素の単価）** | GPT を **§2 の GPT 単価**そのままで提供。**Copilot の"下"で動く基盤**。IaC・VNet・データレジデンシーを自社管理したい企業向け |

> **核心**: **同じ GPT-5 でも、"どの窓口で叩くか"で実効コストが変わります**。Azure OpenAI なら **$1.25/$10** の素の単価。GitHub Copilot 経由なら**モデル乗数でクレジットを消費**し、枠を超えれば追加課金。M365 Copilot なら**座席定額**（重量ユーザーほど割安、ライトユーザーほど割高）。**"トークン単価表"だけでは Copilot のコストは見積もれない**点を、企業の調達・FinOps は押さえる必要があります。

## §5.【本題】フロンティア×安価モデルの役割分担設計

コストを制御する鍵は、**「1つのモデルで全部やらない」**ことです。**難しい判断はフロンティア、定型の量産は安価モデル**に分ける——これがトークンエコノミーの中核設計です。

### どちらが何を担うか

| 観点 | **フロンティア級**（Opus 4.8 / Gemini 3.1 Pro / GPT-5.5 / Fable 5） | **安価級**（Haiku 4.5 / Flash-Lite / GPT-5-nano） |
|:---|:---|:---|
| **得意** | 曖昧性の解消、多段推論、**計画・タスク分解**、**プロンプト/スキーマ設計**、最終検証、難所のデバッグ | 定型実行、大量並列、抽出・分類・整形、要約、単純変換、低レイテンシ処理 |
| **担う工程** | 「**設計図を描く**」「**難しい1件**を解く」「**品質のゲート**」 | 「**設計図どおり量産する**」「**簡単な大量件**をさばく」 |
| **出力単価** | 高（$25〜$50/MTok） | 低（$0.4〜$5/MTok） |
| **原則** | **少ない回数**だけ呼ぶ | **大量に**呼ぶ |

### 実装パターン集

**(a) プロンプト/スキーマ設計はフロンティア → 実行は安価**（最も基本・ご提示の形）
フロンティアに**「曖昧さのない厳密なプロンプト」や「出力スキーマ（JSON 定義）」を1回だけ設計**させ、以降の**量産実行は安価モデル**が担う。例: 契約書1万件からの項目抽出で、抽出ルールとfew-shot例を Opus 4.8 が設計 → 実際の抽出は Haiku 4.5 が1万回実行。**設計コストは定数、実行コストは1/25**。

**(b) ルーター（難易度で振り分け）**
入力を**軽量モデル（または分類器）が難易度判定**し、簡単な案件は安価モデル、難しい案件だけフロンティアへ。カスタマーサポートの一次応答などに有効。

**(c) カスケード（安価→失敗時だけ昇格）**
まず**安価モデルで試行**し、**信頼度が低い/検証に落ちた案件だけフロンティアへエスカレーション**。多くの案件は安価で完結するため、平均単価が下がる。

**(d) オーケストレーション（計画＝フロンティア、実行＝安価並列、統合＝フロンティア）**
フロンティアが**タスクを分解して計画**し、**安価なサブエージェントが並列実行**、最後にフロンティアが**統合・検証**。Claude Code の **Explore エージェントや Dynamic Workflows**（サブエージェントのファンアウト）はまさにこの構造で、[Dynamic Workflows 完全ガイド](/mdTechKnowledge/blog/claude-code-dynamic-workflows-guide/) の考え方が応用できます。

**(e) 生成と検証を分ける（generate-and-verify / adversarial verify）**
**安価モデルが大量生成 → フロンティアが検証/採点**（または逆に、フロンティアが1つの正解を作り安価が横展開）。品質のゲートだけ高いモデルに任せることで、生成コストを抑えつつ品質を担保。

**(f) 蒸留・ゴールデン例で安価を底上げ**
フロンティアで**教師データ（ゴールデン例）**を作り、それを**few-shot / ファインチューニング**で安価モデルに与える。安価モデルの精度を、特定タスクに限ってフロンティア近くまで引き上げる。

**(g) Batch＆キャッシュを安価層に重ねる**
即時性の要らない大量処理は **Batch（−50%）**、繰り返す長い前提文は **プロンプトキャッシュ（入力−最大90%）**。**安価モデル×Batch×キャッシュ**の三段重ねで、実効単価はさらに1桁下がる。

### 削減の概算

> 例: **1万件の定型抽出**タスク。
> - **全部 Opus 4.8**: 1万回 × 高単価 → 仮に **$200/回相当の重い処理なら数千ドル規模**。
> - **設計のみ Opus 4.8（数回）＋実行 Haiku 4.5（1万回）**: 実行単価が **約1/25**。**総額はおおむね1/10〜1/20**に。
>
> 精度が要件を満たす限り、**「計画はフロンティア・実行は安価」に切り替えるだけで一桁のコスト削減**が現実的です（実際の削減率はタスクの検証・エスカレーション率に依存）。

## §6. 企業のコストガードレール（実装）

役割分担を「設計」だけで終わらせず、**組織のガードレール**として仕組み化します。

- **モデル・エンタイトルメント / デフォルトモデル制限**: ロール別に使えるモデルを制限し、既定を安価モデルにする（無自覚なフロンティア濫用を防ぐ）。
- **支出アラート**: 組織 75% / 90%、ユーザー 75% / 95% で通知。→ [Anthropic Enterprise Analytics API 完全ガイド](/mdTechKnowledge/blog/anthropic-enterprise-analytics-api/) を参照。
- **利用実績の可視化・FinOps 連携**: Analytics API を Datadog / CloudZero に流し、**チーム・モデル別の原価**を追跡。
- **Batch / キャッシュの標準化**: 即時不要なジョブは Batch、長い共通前提はキャッシュを"規約"にする。
- **レート上限の把握**: [Rate Limits API 完全ガイド](/mdTechKnowledge/blog/anthropic-rate-limits-api-guide/) でティア・残量をプログラム監視。

## まとめ

- **同じタスクでもモデル次第でコストは100倍以上変わる**。2026年は「フロンティア一択」ではなく、**単価×実トークン数の実効コスト**で設計する時代。
- **Claude / Gemini / OpenAI(GPT)** は per-MTok の素の単価で比較でき、**軽量級は出力単価が 1/10〜1/100**。
- **GitHub Copilot はトークン単価を持たない**（モデル乗数×クレジットのラッパー）、**M365 Copilot は座席課金**、**Azure OpenAI は素のトークン従量**。**"どの窓口で叩くか"で実効コストが変わる**。
- **本質は役割分担**: **正確なプロンプト/スキーマ・計画・検証はフロンティア、量産実行は安価モデル**。ルーター／カスケード／オーケストレーション／Batch・キャッシュを組めば、品質を保ったまま一桁のコスト削減が狙える。
- 企業は**エンタイトルメント・支出アラート・FinOps 連携**でこれを仕組み化する。
