## はじめに — 「どちらもマルチエージェントだが、別物」

Claude Code には複数のエージェントを扱う機能が複数あり、混同されがちです。なかでも **Agent Teams（チームメイト）** と **Dynamic Workflows** は、どちらも「複数のエージェントを動かす」点で似て見えますが、**その本質はまったく異なります**。

よくある疑問が「**この2つは同じ機能を使っているのか？ それとも Dynamic Workflows はサブエージェントの扱いなのか？**」というものです。本記事はこの問いに正面から答えます。結論を先に言えば：

- **Agent Teams** = 対等な**独立セッション**同士が双方向に通信して協働する「チーム」
- **Dynamic Workflows** = 1つの親セッションが**サブエージェント**をコードで束ねて並列実行する「実行エンジン」

つまり **Dynamic Workflows が動かすのはサブエージェント**であり、Agent Teams の「独立セッション」とは別系統です。本記事では両者を1対1で比較し、構造の違いと使い分けを整理します。

> 本記事は2026年6月時点の Claude Code の挙動・公式ドキュメントに基づきます。Agent Teams は実験的機能であり、仕様は変更される可能性があります。

---

## 1. 一目でわかる比較表

| 観点 | Agent Teams（チームメイト） | Dynamic Workflows |
|:---|:---|:---|
| **本質** | 複数の独立セッションが相互通信して協働 | 1セッションがサブエージェントを多数spawnして並列実行 |
| **構成単位** | リーダー＋チームメイト（各々が対等なセッション） | 親 → サブエージェント（従属関係） |
| **動かす実体** | **独立したセッション** | **サブエージェント** |
| **通信方向** | 双方向（DM・共有タスクリスト） | 基本は一方向（親が指示 → 子が結果を返す） |
| **子→親の能動通信** | できる（報告・問い合わせ） | 原則できない（結果テキストを返すだけ） |
| **制御方法** | 各エージェントが自律的に対話 | コード（JavaScript）で決定論的にfan-out/pipeline |
| **規模** | 少数（リーダー＋数体のチームメイト） | 大規模（同時最大16・総計最大1000） |
| **永続性** | セッションとして残り続ける | workflow実行中のみ。終わると結果を返して消える |
| **有効化** | 実験的機能（環境変数で有効化） | 標準機能 |
| **典型用途** | 役割分担した長期協働・相互監視 | 大量タスクの並列処理・fan-out検証 |

## 2. Dynamic Workflows は「サブエージェントの扱い」で正しい

冒頭の疑問「Dynamic Workflows はサブエージェントの扱いか？」——**その理解は正しい**です。

### 2-1. workflow が動かすのはサブエージェント

Dynamic Workflows のスクリプト内で呼び出す `agent(...)` は、すべて**親セッション配下のサブエージェント**を起動します。これは、通常の Agent ツールで spawn するサブエージェントと**同じ「サブエージェント」の系統**です。

その証拠に、Dynamic Workflows のサブエージェントの実行ログは、通常のサブエージェントと同じ場所に保存されます。

```
~/.claude/projects/<project>/<sessionId>/subagents/agent-<agentId>.jsonl
```

つまり「サブエージェント」という共通の土台の上で動いています。

### 2-2. 違いは「起動のされ方」だけ

| 起動方法 | 仕組み |
|:---|:---|
| **通常の Agent ツール** | 人間（or Claude）が手動で1体ずつ起動 |
| **Dynamic Workflows** | JavaScript コードが決定論的に大量・並列で起動（fan-out / pipeline） |

どちらも生み出すのは「サブエージェント」です。Dynamic Workflows は、その**サブエージェントを"コードで束ねて大量並列実行する仕組み"**だと理解すると正確です。サブエージェントは親に従属し、タスクが終わると**結果テキストを親に返して消滅**します。

### 2-3. v2.1.172 以降：サブエージェントは多階層（最大5階層）にネスト可能

当初の Dynamic Workflows はサブエージェントを**フラットにファンアウト**する形が基本でしたが、**Claude Code v2.1.172（2026-06-10 PT）で、サブエージェント自身が自分のサブエージェントを生成できる**ようになりました（公式changelog: *"Sub-agents can now spawn their own sub-agents (up to 5 levels deep)"*、**最大5階層**）。これにより、親 → 子 → 孫…と段階的にタスクを分割する**ツリー型（階層型）の実行**が可能になりました。

ただし、これはあくまで**従属するサブエージェントのツリーが深くなっただけ**であり、Agent Teams の「対等な独立セッションが双方向通信するネットワーク」とは依然として別物です。ネストしても各サブエージェントは親に結果を返して消える点は変わりません（同時最大16・総計最大1000 / run の上限も全体に適用されるため、階層×分岐で延べ数が膨らむ点に注意）。

さらに **v2.1.178（2026-06-15）で `Tool(param:value)` 権限ルール構文**が追加され、例えば `Agent(model:opus)` のように**サブエージェントのモデルを権限ルールで制御（Opus サブエージェントの起動をブロック等）**できるようになりました。大量並列でコストが膨らみやすい Dynamic Workflows のサブエージェントを、モデル単位で抑制できます。

## 3. Agent Teams は「サブエージェント」ではない

一方、Agent Teams のチームメイトは、サブエージェント（親に従属）**ではありません**。それぞれが**独立したセッション**です。この違いが決定的です。

### 3-1. 独立セッションであることの意味

- チームメイトは**自分のコンテキストウィンドウ**を持つ
- リーダーとチームメイトが**双方向にメッセージをやり取り**できる
- リーダーが終了しても、チームメイトのセッションは**別に存在し続ける**

サブエージェントが「親に結果を返したら消える」のに対し、チームメイトは「対等な仲間として生き続け、対話を続ける」——これが両者を分ける核心です。

### 3-2. Agent Teams の3つの連携方法

Agent Teams は、独立セッション同士を次の3つで連携させます。

| 連携方法 | 内容 |
|:---|:---|
| **共有タスクリスト** | 全メンバーがタスクの状態（pending / in progress / completed）を共有 |
| **ダイレクトメッセージング** | メンバーが互いに名前を指定してメッセージを送信 |
| **自動配信** | 送信メッセージは相手に自動的に届く（ポーリング不要） |

「リーダーが実行役チームメイトに5分ごとに状態報告を求め、チームメイトが報告する」といった**相互監視**が組めるのは、この双方向通信があるからです。

## 4. 構造図で理解する

両者の構造を図にすると、違いが視覚的に明確になります。

```
【Dynamic Workflows】サブエージェント方式（従属・一方向・コード制御）

  親セッション
   └─ workflow スクリプト(JS)が制御
        ├─ agent() → サブエージェントA ─┐
        ├─ agent() → サブエージェントB ─┼→ 結果を親に返して消滅
        └─ agent() → サブエージェントC ─┘
        （子は親に従属。子同士・子→親の能動通信なし）


【Agent Teams】独立セッション方式（対等・双方向・自律）

  リーダー session ⇄ チームメイト session A
                   ⇄ チームメイト session B
       （DM・共有タスクリストで相互通信。各々が生き続ける）
```

- Dynamic Workflows は**ツリー構造**（親→子の一方向）
- Agent Teams は**ネットワーク構造**（対等な相互通信）

---

## 5. 重要な共通の制約 — セッションをまたいだ監視はできない

ここは多くの人が誤解する点です。「**すでに別のセッションで走っているタスクを、別のセッションから監視したい**」というニーズはよくありますが——

> **Agent Teams も Dynamic Workflows も、"後から既存の別セッションに割り込んで監視する"ことはできません。**

理由は両者の制約にあります。

| 機能 | なぜ既存別セッションを監視できないか |
|:---|:---|
| **Dynamic Workflows** | サブエージェントは親に閉じている。外部の別セッションを参照できない |
| **Agent Teams** | 「最初から同じチームとして一緒に立てた」メンバー間でのみ通信可能。後から既存の独立セッションには割り込めない |

各 Claude Code セッションは独立したコンテキストを持ち、セッション間を直接つなぐ API は提供されていません。したがって「**既に走っている別セッションの外部監視**」を実現するには、エージェント間通信ではなく、**ファイル経由の監視**（タスク状態ファイルやログを別途チェックする）といった仕組みが必要になります。

---

## 6. どちらを使うべきか

| やりたいこと | 適した機能 |
|:---|:---|
| 大量のタスクを並列で一気に処理したい（fan-out検証・大規模移行など） | **Dynamic Workflows** |
| 役割分担した少数エージェントで長期協働・相互報告させたい | **Agent Teams** |
| 1回の処理で結果だけ欲しい（中間状態は不要） | **Dynamic Workflows**（終われば消える） |
| エージェントを生かしたまま対話を続けたい | **Agent Teams**（セッションが残る） |
| 既に別セッションで走るタスクを外から監視したい | **どちらも不可** → ファイル/ログ監視で対応 |

ざっくり言えば、**「使い捨ての大量並列＝Dynamic Workflows」「生き続ける少数の協働＝Agent Teams」**です。

## まとめ

| ポイント | 結論 |
|:---|:---|
| 2つは同じ機能か？ | **いいえ、別物** |
| Dynamic Workflows の実体は？ | **サブエージェント**をコードで束ねて並列実行する仕組み |
| Agent Teams の実体は？ | 対等な**独立セッション**同士の双方向通信チーム |
| 子→親の能動通信 | Workflows=不可／Teams=可（双方向） |
| 永続性 | Workflows=実行中のみ／Teams=セッションとして残る |
| 既存別セッションの外部監視 | **どちらも不可**（ファイル経由監視が必要） |

「Dynamic Workflows はサブエージェントの扱いか？」という問いの答えは **YES**。Agent Teams のチームメイトとは違い、Dynamic Workflows の `agent()` が生むのは親に従属するサブエージェントです。両者は「複数エージェント」という言葉でひとくくりにされがちですが、**従属するサブエージェントの大量並列（Workflows）か、対等な独立セッションの双方向協働（Teams）か**という、構造レベルで異なる別の道具だと理解するのが正解です。
