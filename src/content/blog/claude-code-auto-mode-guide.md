## はじめに — 「全許可」と「都度確認」の間を埋める

Claude Code を長時間まわしていると、誰もが一度はこう思います。「毎回 **権限確認のプロンプトが出て止まる**のが煩わしい」——。

この承認疲れ（approval fatigue）を一気に消す手段が `--dangerously-skip-permissions` です。名前のとおり**すべてのアクションを無条件で許可**してしまうフラグで、確かに止まらなくなります。しかし「dangerously（危険なほどに）」と冠されているとおり、誤った削除・意図しない外部送信・プロンプトインジェクションへの無防備さといった事故リスクを丸ごと引き受けることになります。

その**安全な代替**として 2026年3月24日に登場したのが **Auto Mode（自律型権限管理）** です。Auto Mode は「すべて止める（都度確認）」と「すべて通す（全許可）」の**中間**に位置し、各アクションを**モデルベースの分類器**で起動前に評価して、安全なものは自動で通し、危ういものだけ人間にエスカレーションします。

> 本記事は公式エンジニアリング記事（[anthropic.com/engineering/claude-code-auto-mode](https://www.anthropic.com/engineering/claude-code-auto-mode)）および InfoQ の報道に基づきます。数値・閾値は公式記載を優先し、公式に明記がない点は「推測」と明示します。なお**リリース日について**は、Issue 起票時点の記録では 2026-03-24 ですが、公式ページの "Published" 表記は March 25, 2026 となっており、表記揺れがあります（タイムゾーン差と推測）。本記事では起票記録に合わせ 2026-03-24 を採用します。

## 1. Auto Mode の位置づけ — `--dangerously-skip-permissions` の安全な代替

従来、Claude Code を「止めずに」走らせたいときの選択肢は大きく2つしかありませんでした。

- **サンドボックス運用**: 安全だが、環境構築・許可リストの整備など**運用コストが高い**
- **`--dangerously-skip-permissions`**: 運用コストはゼロだが、**安全性を放棄**している

公式はこの二者択一を「sandboxing（安全だが高メンテナンス）か `--dangerously-skip-permissions`（ゼロメンテナンスだが危険）か」という構図で説明しており、Auto Mode はそのギャップを埋める**「高い自律性を、低いメンテナンスコストで」（high autonomy at low maintenance cost）**実現するために設計されています。

### 権限モード比較表

Claude Code が持つ主な権限モード（permission mode）を、安全性・自律性・止まりやすさの観点で並べると次のようになります。Auto Mode は「自律性は高いが、危険な操作だけは止まる」という独自のポジションを占めます。

| 権限モード | ファイル編集・変更コマンド | 危険操作のブロック | 承認の出方 | 主な用途 |
|---|---|:---:|---|---|
| **default**（通常） | 都度確認 | 人間が判断 | アクションごとに確認 | 通常作業の標準 |
| **acceptEdits** | 編集は自動承認 | 人間が判断（編集以外） | 編集はスキップ、その他は確認 | 編集を多く伴う作業 |
| **plan**（Plan Mode） | **完全ブロック（read-only）** | そもそも変更不可 | 計画を提示し承認後に実行 | 調査・計画立案 |
| **auto**（Auto Mode） | 分類器が起動前に評価 | **分類器が自動ブロック** | 安全なら自動通過、危ういものだけ確認 | 自律的に走らせたい長時間タスク |
| **dangerously-skip** | 無条件で自動許可 | **なし（全許可）** | 一切確認しない | （非推奨）隔離環境のみ |

ポイントは、Plan Mode が「**変更を一切させないことで安全を確保する**」のに対し、Auto Mode は「**変更を許しつつ、危険な変更だけを分類器で止める**」という、真逆のアプローチで安全と自律を両立している点です。Plan Mode の詳細は [Plan Mode 完全ガイド](/mdTechKnowledge/blog/claude-code-plan-mode-guide/) を参照してください。

## 2. 中核の仕組み — 2段階トランスクリプト分類器

Auto Mode の心臓部は、**Claude Sonnet 4.6 を使ったトランスクリプト分類器（transcript classifier）**です。各アクション（ツール呼び出し）を実行する**前に**、それまでの会話・操作の流れ（トランスクリプト）を分類器に渡し、「ブロックすべきか」を判定します。

ただし、すべてのアクションに対して毎回じっくり推論させると**レイテンシとコストが爆発**します。そこで採用されたのが **2段階構成**です。

### 分類器の2段階表

| 段階 | 何をするか | 速度・コスト | 役割 |
|---|---|---|---|
| **Stage 1: 高速フィルター** | 「ブロックする=yes / しない=no」を**単一トークン**で即断（fast single-token filter）。安全側（caution）に倒す | 高速・低コスト。全アクションに適用 | 大多数の安全な操作を素通しし、怪しいものだけ次段へ |
| **Stage 2: 思考連鎖による精査** | Stage 1 がフラグを立てた場合**のみ**、思考連鎖（chain-of-thought reasoning）で深く検討 | 低速・高コストだが、対象が絞られる | エッジケースの取りこぼし（recall）を救い、誤検知を削る |

この「**速い門番（全件）＋ 慎重な審査官（疑わしい件のみ）**」の組み合わせにより、レイテンシと計算コストを抑えつつ、判定精度を引き上げています。InfoQ も「効率とカバレッジのバランスを取るための two-stage classification approach」と報じています。

### 精度の実測値（公式）

公式記事は、実トラフィック（n=10,000）に対する誤判定率を次のように示しています。数値は「危険な操作をどれだけ取りこぼすか／安全な操作をどれだけ誤ってブロックするか」のトレードオフを表します。

| 指標 | Stage 1 単体 | 2段階パイプライン全体 |
|---|---|---|
| **偽陽性率（FPR・安全な操作を誤ブロック）** | 8.5% | **0.4%** |
| **偽陰性率（FNR・危険な操作を見逃し）** | 低め | 約 **17%**（過剰なアクションの取りこぼし） |

つまり 2段階化によって「安全な操作を誤って止めてしまう率」を 8.5% → 0.4% へ大幅に下げる一方、「本当に危ない操作を見逃す率」は約17%残るという結果です。**完璧な判断ではない**ため、Auto Mode は「無人で完全放置してよい」ものではなく、後述のエスカレーション停止や `.claudeignore` と**組み合わせて**安全域を確保する設計になっています。

> **補足（出力層・入力層の多層防御）**: 公式は分類器（出力層の防御）に加えて、ツール出力をエージェントが処理する前に検査する **server-side prompt-injection probe（プロンプトインジェクション探知）** にも言及しています。Auto Mode 単独ではなく、入力・実行の各レイヤーで多層的に守る構成です。

## 3. エスカレーション停止 — 連続3拒否 / 累計20拒否で自動停止

分類器がどれだけ優秀でも、判断が揺れているセッションを延々と走らせるのは危険です。そこで Auto Mode には **バックストップ（最終的な歯止め）** が組み込まれています。

公式の表現では「セッションが **連続3回の拒否（3 consecutive denials）** または **累計20回の拒否（20 total）** に達した場合」、エージェントは**自動的に停止し、人間のレビューにエスカレーション**します。

### 停止条件表

| 条件 | 閾値 | 意味 | 起きたときの挙動 |
|---|---|---|---|
| **連続拒否** | 3回 連続 | 短時間に立て続けに弾かれている＝方向性がズレている兆候 | エージェントを停止し人間に判断を委ねる |
| **累計拒否** | 20回 累計 | セッション全体で拒否が積み上がっている＝リスクの高い作業 | エージェントを停止し人間に判断を委ねる |

この「連続」と「累計」の二重のカウンタにより、**「一時的に詰まっただけ」と「そもそも危険な作業を繰り返している」**を区別しつつ、暴走を確実に止めます。`--dangerously-skip-permissions` にはこうした歯止めが**一切ない**点が、Auto Mode との決定的な差です。

## 4. `.claudeignore` 連携 — 触れてほしくない場所を宣言する

Auto Mode は分類器による動的な判定に加えて、**`.claudeignore`** による静的な除外指定とも連携します。`.gitignore` と同様の記法で、Claude に**触れさせたくないファイル・ディレクトリ**を宣言しておく仕組みです。

| 仕組み | 性質 | 役割 |
|---|---|---|
| **2段階分類器** | 動的（その場の文脈で判定） | アクションごとに危険性を評価 |
| **`.claudeignore`** | 静的（事前に宣言） | 機密ファイル・秘密情報などを最初から対象外にする |

たとえば `.env`、認証情報、本番設定、巨大な生成物などを `.claudeignore` に列挙しておけば、Auto Mode で自律的に走らせていても、それらの領域には踏み込ませない**事前の防壁**になります。動的判定（分類器）と静的宣言（`.claudeignore`）を**併用**することが、Auto Mode を安心して使う前提条件と言えます。

> **推測**: `.claudeignore` の記法詳細（ネガティブパターンの可否、グロブの厳密な解釈など）は本記事執筆時点の公開情報では網羅されていません。実運用前に最新の公式ドキュメントで挙動を確認することを推奨します（この段落は推測を含みます）。

## 5. サブエージェント起動も「起動前」に評価される（v2.1.178）

Auto Mode の安全性は、その後のバージョンアップでさらに強化されています。2026年6月15日（PT）の **Claude Code v2.1.178** で、次の変更が入りました。

> Improved auto mode: subagent spawns are now evaluated by the classifier before launch（公式 changelog）

これにより、**サブエージェントの起動が「起動前」に分類器で評価**されるようになり、「レビューを経ずに起動したサブエージェントが、許可されていないアクションを要求できてしまう抜け穴」が塞がれました。Auto Mode は単なる「ツール1回ごとのフィルター」ではなく、**委任の階層まで含めて**安全を担保する方向に進化しています。サブエージェントの権限分離そのものについては [サブエージェントの歩き方](/mdTechKnowledge/blog/claude-code-subagent-readonly-switch/) で詳しく扱っています。

## 5.5. 【2026-06-25】v2.1.193 — 全シェルコマンドの分類と拒否理由の可視化

2026年6月25日（PT）の **Claude Code v2.1.193** で、Auto Mode の安全分類まわりに2つの実用的な変更が入りました。

| 設定 / 変更 | 内容 | 既定 |
|---|---|---|
| **`autoMode.classifyAllShell`** | **すべての Bash / PowerShell コマンド**を Auto Mode 分類器に通す。従来は **任意コード実行（arbitrary-code-execution）パターンのみ**が分類対象だったため、本設定でカバレッジが広がる（その分、分類器の呼び出し回数は増える） | **オプトイン**（既定は従来動作） |
| **拒否理由の表示** | Auto Mode が拒否した際の **denial reasons** を、**トランスクリプト・拒否トースト・`/permissions` の「最近の拒否」一覧**に表示 | v2.1.193 以降の標準動作 |

`classifyAllShell` は「中核の仕組み（2段階分類器）」をシェル全般へ拡張するオプションで、**より厳密に倒したい運用**向けです。拒否理由の表示は、「なぜブロックされたのか」を後から追跡しやすくし、Auto Mode の挙動をチューニングする際のデバッグ性を高めます。

参考: [Claude Code v2.1.193 リリースノート](https://github.com/anthropics/claude-code/releases/tag/v2.1.193) ／ [Anthropic Engineering: Claude Code Auto Mode](https://www.anthropic.com/engineering/claude-code-auto-mode)

## 6. いつ Auto Mode を選ぶか — 卒業の判断基準

`--dangerously-skip-permissions` を使っている人が Auto Mode へ「卒業」すべきかは、次の観点で判断できます。

| 状況 | 推奨モード | 理由 |
|---|---|---|
| 隔離されたコンテナ・使い捨て環境で短時間 | dangerously-skip でも可 | 失っても困らない環境なら歯止めは不要 |
| ローカルの実プロジェクトを自律的に走らせたい | **auto** | 危険操作だけ止まり、承認疲れも減る |
| 影響範囲が読めない大改修の前 | plan → 承認 → auto | まず計画を固めてから自律実行へ移す |
| 機密ファイルが混在するリポジトリ | **auto ＋ `.claudeignore`** | 静的除外で踏み込ませない壁を併用 |

実務的なおすすめは、**「`--dangerously-skip-permissions` の常用をやめ、Auto Mode ＋ `.claudeignore` を既定にする」**ことです。約17%の偽陰性が残るとはいえ、連続3拒否・累計20拒否の自動停止という歯止めがある分、無防備な全許可よりはるかに安全側です。

## まとめ

- **Auto Mode は `--dangerously-skip-permissions` の安全な代替**として 2026-03-24 に登場した中間モード
- 中核は **Sonnet 4.6 による2段階トランスクリプト分類器**（高速フィルター ＋ フラグ時のみ思考連鎖）で、FPR を 8.5% → 0.4% に削減
- ただし **偽陰性は約17%残る**ため完璧ではなく、**連続3拒否 / 累計20拒否で自動停止**するバックストップを併用
- **`.claudeignore`** による静的な除外宣言と組み合わせることで、動的判定の穴を補う
- v2.1.178 で**サブエージェント起動も起動前評価**の対象になり、安全性がさらに強化された
- 結論として、隔離環境以外では **`--dangerously-skip-permissions` を卒業し、Auto Mode ＋ `.claudeignore` を既定にする**のが安全と自律の最良バランス
