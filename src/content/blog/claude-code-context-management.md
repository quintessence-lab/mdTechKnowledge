# Claude Codeのコンテキスト管理入門 — /compact・/clear・メモリ・セッション復元の使い分け

> 2026年4月時点

「コンテキストってよく聞くけど、何を指しているのかが曖昧」「`/compact`を実行するとトークンを消費するの？」「端末を再起動したら全部消えてしまう？」「`/clear`は部分的に消せる？」 — Claude Codeを日常的に使い始めると、こういった疑問に必ず行き当たります。本記事ではコンテキストの基礎から、運用上の使い分けまでを初心者にも分かるように整理します。

---

## 1. そもそも「コンテキスト」とは？

**コンテキストとは、Claude（モデル）が一度の応答を生成するときに参照する「全ての情報」のこと**です。具体的には以下が含まれます。

| 含まれるもの | 例 |
|------------|-----|
| システムプロンプト | Claude Code本体が組み込む基本指示（ツール定義、応答ルール） |
| グローバル設定 | `C:\Users\quint\.claude\CLAUDE.md`（全プロジェクト共通の指示） |
| プロジェクト設定 | カレントディレクトリの `CLAUDE.md` |
| メモリ | `MEMORY.md` および参照先の各メモリファイル |
| 会話履歴 | 過去のユーザー発言・Claude応答・ツール実行結果 |
| 添付ファイル | 直近のセッションで `Read` したファイル内容 |
| 環境情報 | OS、シェル、作業ディレクトリ、現在日付 |

これらを毎回まとめて1つの長い入力としてClaudeに渡しています。会話が長くなるほどコンテキストは膨らみ、**上限（200K〜1Mトークン）に近づくと古い情報が圧縮・破棄**されます。

### コンテキストとモデルの関係

| 観点 | 内容 |
|------|------|
| 単位 | トークン（≒英単語1.3個分、日本語は約1.5文字で1トークン） |
| 保持期間 | セッション中のみ。セッション終了で揮発（ただしJSONLログには保存） |
| 課金 | 会話のたびにコンテキスト全体が「入力トークン」として課金対象 |
| キャッシュ | プロンプトキャッシュにより、同じ前半部分は値引き（5分TTL） |

## 2. ターミナル間・Claude.aiとのコンテキスト共有

**結論: 基本的に共有できません。**

- 各Claude Codeセッション（ターミナル）は**独立したコンテキスト**を保持します
- 別ターミナルを開けば別セッションとして開始
- Claude.ai（Web版）とClaude Code（CLI版）は**完全に別システム**で、コンテキスト共有はありません

### ただし、間接的に情報を引き継ぐ手段はあります

| 手段 | 仕組み | 用途 |
|------|--------|------|
| **メモリシステム** | `~/.claude/projects/*/memory/` に保存、全セッションで自動読込 | ユーザー設定・好み・恒久的なルール |
| **CLAUDE.md** | グローバル/プロジェクト設定として全セッション読込 | プロジェクト固有の運用ルール |
| **ファイルベース引き継ぎ** | 作業内容をmdに書いて次セッションで `Read` | 進行中タスクの引き継ぎ |
| **MCPサーバ経由** | 外部システム（GitHub、DB等）に状態を保存 | 複数環境で同じ情報を参照 |

> 💡 **ポイント**: 「セッションを跨ぐ情報」と「セッション内の情報」を意識して設計することが運用品質に直結します。

## 3. /compactとトークン消費の関係

**はい、`/compact`実行時もトークン制限枠を消費します。**

### /compactの内部動作

1. `/compact`コマンドを実行すると、Claude Code内部で「会話履歴を要約してください」というLLM呼び出しが走る
2. **入力**: 圧縮前の全会話履歴（数万〜数十万トークン）
3. **出力**: 要約テキスト（数千トークン）
4. 入力 + 出力の両方が**通常の課金対象**かつ**5時間レート制限のカウント対象**

### 引数による制御

| コマンド | 動作 |
|---------|------|
| `/compact` | デフォルトで全体を要約 |
| `/compact 実装内容のみ` | 「実装内容」に重点を置いた要約（不要情報は省かれやすい） |
| `/compact <自由テキスト>` | 任意の指示で要約方針を制御 |

### /compactのトレードオフ

| メリット | デメリット |
|---------|-----------|
| 以降の会話で長期的にトークン節約 | 1回の実行自体は重い処理（時間 + トークン） |
| 上限超過によるエラーを回避 | 細かいニュアンスや具体的なコードが消える可能性 |
| キャッシュの再構築（5分TTL）でも安定運用可 | キャッシュは無効化され、次回応答は通常コスト |

### 自動 /compact

コンテキストが上限の80%程度に近づくと、Claude Codeが**自動的に /compact を起動**します。手動で早めに実行する場合と異なり、タイミングは選べないため、計画的に進めたい場合は手動実行が推奨されます。

## 4. 端末再起動後のセッション復元

**再起動でメモリ上のコンテキストは初期化されますが、ディスク上のJSONLログから復元できます。**

### 自動保存の仕組み

- すべてのセッションは `C:\Users\quint\.claude\projects\C--Users-quint-clauderc\` に**JSONLログ**として記録
- 例: `509b2925-f0c6-48f4-a89c-c2f1882b974a.jsonl`
- ファイル名はセッションIDで、1セッション=1ファイル
- ツール実行結果やメッセージ全文を含む完全な履歴

### 復元コマンド

| コマンド | 動作 |
|---------|------|
| `claude --resume` | 最近のセッション一覧を表示し、選んで再開 |
| `claude -c` | 直前のセッションを継続 |
| `claude --resume <session-id>` | 特定セッションを指定して再開 |

### 定期的に必要情報を保存する方法

セッションが揮発しても困らないよう、重要情報は以下のいずれかに**昇格**させておきます。

| 保存先 | 適している情報 |
|--------|--------------|
| `MEMORY.md` + 個別メモリファイル | ユーザー設定、好み、フィードバック、頻繁に参照するルール |
| `CLAUDE.md`（グローバル/プロジェクト） | プロジェクト固有の運用ルール、コーディング原則 |
| ハンドオフ用md（プロジェクト配下） | 進行中タスクの状態、次回やること |
| GitHub Issue / プロジェクト管理 | チームで共有が必要な作業 |

> 💡 **コツ**: 「同じ情報を2回以上説明したな」と思ったら、それはメモリ昇格のサインです。

## 5. /clearコマンドの動作

**`/clear`は全削除のみで、部分削除機能はありません。**

### /clear vs /compact の比較

| 項目 | `/clear` | `/compact` |
|------|---------|------------|
| 履歴の扱い | **完全消去**（ゼロから） | **要約に圧縮**（情報は残る） |
| トークン消費 | なし（即時実行） | あり（要約LLM呼び出し） |
| メモリ・CLAUDE.md | 影響なし（自動再読込） | 影響なし |
| 作業継続性 | 文脈は失われる | 文脈は粒度が荒くなりつつ残る |
| 用途 | 全く別の作業を始めるとき | 同じ作業を続けたいがコンテキストが重いとき |

### 部分的に消したい場合の代替手段

`/clear`に部分削除機能はないため、以下で代用します。

1. **`/compact <focus>`**: 残したい内容を指示することで、不要部分が自然に省かれる
   - 例: `/compact 実装したコードと現在のタスク状態のみ`
2. **メモリ編集**: `MEMORY.md` および個別メモリファイルを直接編集
3. **新セッション開始 + 必要情報のみ引き継ぎ**: `/clear` 後、必要なファイルのみ `Read` で読み込む

## 6. 実用的な使い分けマトリクス

| 状況 | 推奨アクション | 理由 |
|------|--------------|------|
| トークン残量が少ないが作業を続けたい | `/compact` | 文脈を保ったまま圧縮 |
| 全く別の作業を始めたい | `/clear` | 不要な前提を完全排除 |
| 長期保存したい知見を得た | `MEMORY.md`へ追記 | 全セッションで再利用可 |
| 別端末で続きをしたい | `MEMORY.md` + ハンドオフmdに記録 | ファイル経由で間接共有 |
| 過去セッションを復元したい | `claude --resume` | JSONLログから完全復元 |
| 並列で別タスクをこなしたい | 別ターミナル + サブエージェント | コンテキスト独立で干渉なし |

## 7. コンテキスト節約のベストプラクティス

### 7-1. サブエージェント活用

調査系・探索系のタスクは `Agent` ツールに委任することで、メインコンテキストの消費を最小限に抑えられます。サブエージェントは別コンテキストで実行され、最終結果のみメインに返ります。

```
例: 「コードベースを調査してXの実装箇所を特定して」
→ Agent(subagent_type="Explore") に委任
→ メインには「該当ファイル: src/foo.ts:42」のみ返却
```

### 7-2. ファイル全読みの回避

`Read` の `offset` / `limit` パラメータで必要箇所のみ取得します。

```
Read(file_path="huge.md", offset=100, limit=50)
```

### 7-3. 早めの /compact

上限80%を超える前に手動実行すると、要約品質が安定します。自動圧縮は急に発火するため、計画的な作業を阻害することがあります。

### 7-4. メモリへの昇格

「何度も同じ説明をしている」「次回も覚えていてほしい」と思ったら、即座にメモリへ移します。会話履歴に残すよりも、メモリに固定したほうが**読み込みコスト**も**精度**も上がります。

### 7-5. ツール出力の制御

`Grep` の `head_limit` や `Bash` の `head -N` で、長大な出力を絞り込みます。コンテキストに残るのは最終的にツール結果なので、出力サイズ管理は重要です。

## 8. よくある誤解

| 誤解 | 正しい理解 |
|------|-----------|
| 「ターミナルを2つ開けば共通のコンテキストで作業できる」 | 各ターミナルは独立。共有したい情報はメモリかファイルへ |
| 「`/compact`は無料の機能」 | 内部でLLM呼び出しが走るためトークン消費あり |
| 「再起動するとすべて消える」 | JSONLログがディスクに残る。`claude --resume` で復元可 |
| 「`/clear`で特定の発言だけ消せる」 | 全削除のみ。部分削除は `/compact <focus>` で代用 |
| 「メモリを書いておけば毎回詳細まで思い出せる」 | メモリは要約として読まれる。詳細はソースを参照する設計が安全 |

---

## 9. まとめ

- **コンテキスト**は「Claudeが応答時に参照する全情報」。会話履歴・設定・メモリ・添付ファイルを含む
- **セッション間の共有**は基本的にできない。メモリ・CLAUDE.md・ファイル経由で間接共有する
- **`/compact`はトークン消費あり**。レート制限にもカウントされるが、長期的には節約になる
- **再起動後の復元**は `claude --resume` でJSONLログから可能
- **`/clear`は全削除のみ**。部分削除は `/compact <focus>` を活用
- **長期的な情報**はメモリやCLAUDE.mdへ昇格させる運用が効率的

Claude Codeを快適に使い続ける鍵は、**「セッション内に残すもの」と「セッションを跨いで残すもの」を意識的に分離する**ことです。本記事のマトリクスを目安に、自分の運用にフィットする使い分けを見つけてください。
