## はじめに — 「いきなり書き換えられる」不安をなくす

Claude Code に大きな変更を頼むとき、こんな不安はないでしょうか。「**意図と違う方向にコードを書き換えられたらどうしよう**」「影響範囲が分からないまま実行されると怖い」——。

この不安に答えるのが **Plan Mode（プランモード）** です。Plan Mode は、**ファイルを一切変更せず、コードベースの調査と"これからどう変更するか"の計画立案だけ**を行うモードです。Claude は計画を提示し、**あなたが承認して初めて実行に移ります**。いわば「設計図を見せてもらってから工事を始める」進め方です。

本記事は、Plan Mode の入り方・抜け方、読み取り専用の制約、計画の承認フロー、他の権限モード（permission mode）との違い、そして実務での使いどころまでを、体系的に解説する完全ガイドです。

> 本記事は2026年5月時点の Claude Code の挙動・公式ドキュメントに基づきます。キー操作や選択肢の表記はバージョン・環境により多少異なる場合があるため、不確実な点は「要確認」と明記しています。

---

## 1. Plan Mode とは — 目的と位置づけ

Plan Mode は、Claude Code が持つ複数の **Permission Mode（権限モード）** の一つです。その役割は明確で、**「読み取り専用（read-only）で調査し、変更方針を提案し、実行前にユーザー承認を得る」**ことに特化しています。

| 観点 | 通常モード（default） | Plan Mode |
|:---|:---|:---|
| ファイル読取・調査 | 自動 | 自動 |
| ファイル編集・作成 | 都度確認 | **不可（完全ブロック）** |
| 変更系コマンド実行 | 都度確認 | **不可** |
| 計画の提示 | なし | **あり（承認フロー付き）** |
| 承認の単位 | アクションごと | **計画全体を一度に** |

ポイントは、Plan Mode が「アクションごとに確認する」のではなく、**「まず全体の計画を見せ、合意してからまとめて実行する」**という流れを作ることです。これにより、調査の途中で勝手にファイルが書き換わる心配がなく、方針のズレを実行前に正せます。

---

## 2. Plan Mode への入り方・抜け方

### 2-1. 入り方

**方法1：Shift+Tab でモードを循環（最も手軽）**

Claude Code のプロンプトで `Shift+Tab` を押すたびに、権限モードが循環します。標準的な順序は次のとおりです（環境・設定により表示されるモードは変動します）。

```
default（都度確認）
   ↓ Shift+Tab
acceptEdits（編集を自動承認）
   ↓ Shift+Tab
plan（プランモード）★
   ↓ Shift+Tab
（auto / bypassPermissions が条件次第で表示）
```

`plan` と表示された状態が Plan Mode です。

**方法2：起動時にフラグ指定**

```bash
claude --permission-mode plan
```

最初から Plan Mode で起動したいときはこれが確実です。

**方法3：デフォルトに設定**

`settings.json`（`~/.claude/settings.json` またはプロジェクトの `.claude/settings.json`）で、起動時のデフォルトを Plan Mode にできます。

```json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

> `/plan <プロンプト>` のように、その場の1プロンプトだけ Plan Mode で実行する使い方ができるバージョンもあります（挙動はバージョン依存・要確認）。

### 2-2. 抜け方

| やりたいこと | 操作 |
|:---|:---|
| 計画を承認せず通常モードへ戻る | もう一度 `Shift+Tab` でモードを循環（立てた計画は破棄） |
| 計画を承認して実行へ移る | Claude が提示する承認メニューから選択（次章） |

## 3. 計画の承認フロー — Plan Mode の核心

Plan Mode で調査が終わると、Claude は「**こう変更します**」という計画を提示し、進め方を尋ねてきます。代表的な選択肢は次のとおりです（表記はバージョンにより異なる場合あり）。

| 選択肢 | 意味 | 承認後のモード |
|:---|:---|:---|
| **計画を承認し、自動で実行** | 計画に同意し、以降は自動実行で進める | auto（条件を満たす場合） |
| **計画を承認し、編集は自動・確認は事後** | ファイル編集を自動で行い、結果を後でまとめて確認 | acceptEdits |
| **計画を承認し、各編集を都度確認** | 一手ずつ確認しながら慎重に進める | default |
| **計画を続ける（フィードバック）** | 承認せず、要望を伝えて計画を練り直す | plan のまま |
| **Ultraplan で精緻化** | より高度な計画レビューへ（後述） | — |

つまり Plan Mode は「計画づくり」と「実行モードの選択」を**1つの承認ステップに統合**しています。承認時に「どれくらい自動で進めてよいか」まで一緒に決められるのが特徴です。

> **ヒント**: 影響範囲が読めない作業では「各編集を都度確認（default）」を、方針が固く単純作業なら「自動で実行（auto/acceptEdits）」を選ぶと、安全性と速度のバランスを取れます。

---

## 4. Plan Mode 中の制約 — できること／できないこと

Plan Mode の安全性は「**何ができないか**」が明確に決まっていることに支えられています。

### ✅ できること（自動実行）
- ファイル読取（Read）、ファイル検索（Glob / Grep）
- 読み取り専用のシェルコマンド（`ls` `cat` `find` `grep` `pwd` など）
- コードベースの構造把握・調査

### ⚠️ 都度確認が必要
- 書き込み系シェルコマンド（`mkdir` `touch` `rm` `mv` `cp` `sed` など）
- ネットワークリクエスト（WebFetch など）

### ❌ できないこと（完全にブロック）
- **ファイルの編集・作成**（Write / Edit）
- **ビルド・テストの実行**（`npm run` `pytest` など状態を変える実行）
- **git commit / push**
- その他すべての破壊的・状態変更操作

この「読むのは自由、変えるのは承認後」という線引きが、Plan Mode を**安心して任せられる調査モード**にしています。

## 5. 他の Permission Mode との比較

Plan Mode の位置づけを理解するには、他のモードと並べるのが早道です。

| モード | 読取 | ファイル編集 | 破壊的コマンド | 特徴 |
|:---|:---:|:---:|:---:|:---|
| **default** | ✓ | 都度確認 | 都度確認 | 最も安全。すべて確認 |
| **plan** | ✓ | **✗（不可）** | ✗ | 読み取り専用＋計画提示。**設計フェーズ専用** |
| **acceptEdits** | ✓ | ✓（自動） | ✗ | 編集は自動、破壊的操作は確認 |
| **auto** | ✓ | ✓ | △（分類器が判定） | 安全判定後に自動実行。速度重視 |
| **bypassPermissions** | ✓ | ✓ | ✓ | 一切確認しない。**隔離環境専用** |

- **plan ↔ acceptEdits/auto** は対極です。plan は「実行前に止まって計画を見せる」、後者は「止まらず進める」。
- **plan は唯一、編集そのものを禁止**するモードです。他は程度の差こそあれ編集を許します。

## 6. 使いどころ — どんなときに Plan Mode を使うか

| シーン | 理由 |
|:---|:---|
| **大規模リファクタリングの前** | 複数ファイルにまたがる変更方針を、着手前に見える化して合意したい |
| **未知のコードベースの調査** | 構造を理解したうえで変更戦略を立て、的外れな実装を防ぐ |
| **影響範囲が不明な作業**（削除・移行・置換） | 「どこに波及するか」を先に洗い出してから実行したい |
| **レビュー必須の環境** | コンプライアンス上、変更内容の事前承認が求められる |
| **方針が固まっていない段階** | まず相談・計画し、納得してから実装に入りたい |

### 実務フローの例

```
1. claude --permission-mode plan で起動
2. 「ユーザー削除時に関連レコードが消えないバグを直したい」と指示
3. Claude が読み取り専用で調査
   - DBスキーマを読む / 関連コードを grep / 原因を特定
4. 計画を提示
   「comments・likes・notifications に CASCADE 削除を設定。
    変更対象: migrations/xxx.sql」
5. ユーザーが「各編集を都度確認」で承認
6. default モードで一手ずつ確認しながら実装 → テスト → コミット
```

「**いきなり実装させず、まず方針を固める**」——これが Plan Mode の王道の使い方です。

## 7. Ultraplan・Dynamic Workflows との関係

Plan Mode は、関連する2つの機能と組み合わせて理解すると、立ち位置がより明確になります。

| 機能 | 役割 | Plan Mode との関係 |
|:---|:---|:---|
| **Ultraplan** | ブラウザベースの高度な計画レビュー | Plan Mode の計画を、より精緻にレビュー・改善する発展版 |
| **Dynamic Workflows** | 並列サブエージェントによる大規模実行 | 「計画して止まる」Plan Mode に対し「自動で並列実行する」対極の機能 |

- **Ultraplan** は Plan Mode の計画づくりをさらに強化したい場合の選択肢です（詳細は別記事「[Ultraplan ガイド](/mdTechKnowledge/blog/ultraplan-guide/)」）。
- **Dynamic Workflows** は「正確さ vs 速度・規模」という観点で Plan Mode としばしば比較されます。どちらが精度面で適するかは、別記事「[Plan Mode と Dynamic Workflows、精度が高いのはどっち？](/mdTechKnowledge/blog/claude-code-plan-vs-workflow-accuracy/)」で詳しく扱っています。

## まとめ

| 観点 | ポイント |
|:---|:---|
| **何のモードか** | 読み取り専用で調査・計画し、承認後に実行へ移る安全モード |
| **入り方** | `Shift+Tab` で循環／`--permission-mode plan`／`settings.json` の `defaultMode` |
| **制約** | 読取・調査は自由、**編集・ビルド・commitは不可**（承認後に解禁） |
| **承認フロー** | 計画提示と同時に「どの実行モードで進めるか」を選択できる |
| **使いどころ** | 大規模変更前・未知コードベース・影響範囲不明・レビュー必須環境 |
| **関連** | 計画強化＝Ultraplan／並列実行＝Dynamic Workflows |

Plan Mode は、Claude Code を「**勝手に進む道具**」から「**方針を合意してから動く相棒**」に変える機能です。とくに重要な変更ほど、いきなり実装させるのではなく、まず Plan Mode で計画を固める——この一手間が、手戻りと事故を大きく減らします。`Shift+Tab` ひとつで入れるので、大きな作業の前にはまず Plan Mode、を習慣にしてみてください。
