## はじめに — フラットフィー時代の終わり

2026年4月16日、The Register が「Anthropic ejects bundled tokens from enterprise plans」と報じたことで、Anthropic のエンタープライズ料金体系の大規模な見直しが広く知られることになりました。報道のポイントを一文で要約すると、こうなります。

> シート料金にトークン使用量があらかじめ含まれる「バンドル型」プランを廃止し、シート料金は「プラットフォームへのアクセス権」のみを意味する完全分離モデルへ移行した。

これは単なる価格表の差し替えではなく、Anthropic がエンタープライズ顧客との契約構造そのものを再設計した、というシグナルです。同様の流れは OpenAI の Codex、GitHub の Copilot、Windsurf などにも見られ、生成 AI 業界全体で「フラットフィー時代」が終わりつつある状況が浮かび上がります。

本記事では、Anthropic の料金体系移行を以下の観点で整理します。

- 旧プラン（バンドル型）と新プラン（完全分離型）の構造比較
- 2025年11月から2026年3月8日までの移行タイムライン
- 軽量利用者・重量API利用者への影響の非対称性
- 旧契約の更新タイミングと猶予の扱い
- 業界全体の従量課金シフトとの関係

なお、本記事は公開報道情報に基づくものであり、契約条件は組織ごとに異なります。実際の契約交渉では Anthropic 側の最新提示を必ず確認してください。

## 旧プランの構造 — Chat-only / Standard / Premium

2025年秋まで Anthropic がエンタープライズ向けに提供していたシート型プランは、おおむね次の3階層に整理できます。

| プラン名 | 主な対象 | バンドル内容 |
| --- | --- | --- |
| Chat-only | claude.ai を社内利用するチーム | Claude.ai チャット利用枠 |
| Standard | Claude API も併用するチーム | チャット + 一定量のAPIトークン |
| Premium | 重量級ユーザー・大規模開発組織 | チャット + より大きなAPIトークン枠 + Claude Code 利用枠 |

各プランは「シート単価 × 人数」というシンプルな課金で、シート料金の中に**事前に定義されたトークン量**が含まれていました。これを超えた分はオーバーエージ（超過課金）として API 標準レートで請求されます。

このモデルには、エンタープライズ側にとって次のような利点がありました。

- **予算化が簡単**: 月額がほぼ固定で、CFO・調達部門が承認しやすい
- **個人差が均される**: チーム内のヘビーユーザーとライトユーザーがバンドル内で相殺される
- **採用ハードルが低い**: 「とりあえず10シート」といった導入が可能

一方で、Anthropic 側には次のような課題がありました。

- **オーバーエージが収益の主軸になる組織が増えた**: 一部のコンサルティングファームでは「シート料金は20%、超過課金が80%」という比率が常態化していた
- **キャパシティ計画が難しい**: バンドルトークンの未消化分も含めて GPU/TPU 容量を確保する必要があった
- **モデル改良に伴うトークン消費増にバンドル枠が追従しにくい**: 推論モデルやエージェント機能の普及で、想定を上回る消費が頻発

## 新プランの構造 — シートとトークンの完全分離

2026年4月時点で Anthropic が提示している新しいエンタープライズ料金は、次のように要約できます。

```
請求額 = （シート単価 × 人数）+ Σ（モデル別トークン消費 × API 標準レート）
```

報道によれば、シート単価は **1人あたり月額20ドル**を軸とした単一階層に集約され、ここには**Claude.ai プラットフォームへのアクセス権**のみが含まれます。Claude（チャット）、Claude Code、Claude Cowork のどれを使った場合も、トークン消費は API 標準レートで個別に計上されます。

Anthropic 自身は次のように説明したと報じられています。

> The seat fee only covers access to the platform and doesn't include any usage. All usage across Claude, Claude Code, and Cowork is billed separately at standard API rates, based on what your team actually consumes.

新旧モデルを図解すると次のようになります。

```
旧プラン（バンドル型）
  ┌─────────────────────────────┐
  │  シート料金                  │
  │  ┌──────────────────┐       │
  │  │ バンドルトークン │       │
  │  └──────────────────┘       │
  └─────────────────────────────┘
         + オーバーエージ（API標準レート）

新プラン（完全分離型）
  ┌──────────────┐   ┌──────────────────────────────┐
  │ シート料金   │ + │ 全トークン消費（API標準レート）│
  │ （アクセス権）│   │  Claude / Claude Code / Cowork │
  └──────────────┘   └──────────────────────────────┘
```

新モデルの本質は、**「アクセス権」と「使用量」を完全に切り離した**点にあります。これにより、シート料金は限りなく「ライセンスフィー」に近い性格となり、モデル消費の変動はすべて従量側に集約されます。

## 移行タイムライン

報道情報をもとに移行の時系列を整理すると、次のとおりです。

| 時期 | 内容 |
| --- | --- |
| 2025年11月 | 新規契約および更新案件で usage-based プランへの切り替えを開始 |
| 2026年2月 | 新エンタープライズシート（月額20ドル/人を軸とする単一階層）を正式導入 |
| 2026年3月8日 | ドキュメントを更新し「Chat-only / Standard / Premium シートは新規契約では利用不可」と明示 |
| 2026年4月16日 | The Register が一連の変更をまとめて報道、業界に広く認知される |

ポイントは、**2025年11月から段階的に切り替わっていた**ことです。新規契約や契約更新のタイミングを迎えた組織は、知らずのうちに新モデルに移行していたケースも珍しくありません。

旧プランを継続しているグランドファザード（既存契約の維持）顧客であっても、契約更新のタイミングで新モデルへの移行が前提となります。固定のバンドル枠を維持し続けたい場合、Anthropic 側との個別交渉が必要になり、すべての組織で受け入れられるとは限りません。

## 軽量利用者・重量利用者それぞれの影響

新モデルの導入は、組織の利用パターンによって**正反対の経済的インパクト**をもたらします。

### 軽量利用者（コスト増）

シート枠の中で「とりあえず Claude にアクセスできる人を確保する」ような使い方をしていた組織は、新モデル下では次のような問題に直面します。

- 旧プランでは「使わなくてもバンドル分は無駄になるだけ」だったが、新プランでは消費した分だけ追加請求される
- ライトユーザー数十名のうち、月のうち数回しか使わない人もカウントすると、シート料金とトークン料金の合計が旧プランを上回る可能性がある
- 部門単位でアカウントをまとめて発行していた場合、利用実態と請求が一致しなくなる

### 重量 API 利用者（影響軽微〜中立）

すでにバンドル枠を超過し、超過課金が請求の大半を占めていた組織は、影響が軽微になる傾向があります。あるコンサルティングファームのケースでは「シート料金が20%、オーバーエージが80%」という比率だったため、新モデルでも実質的な総額がほぼ変わらなかったと報じられています。

### Claude Code を本格運用しているチーム

Claude Code を CI/CD・社内ツールから呼び出しているチームは、エージェント実行やバックグラウンド処理に伴うトークン消費が大きくなる傾向があります。新モデル下では、この消費がそのまま月次請求に反映されるため、**FinOps（クラウドコスト最適化）と同等のトークンコスト管理プラクティス**が必要になります。

利用パターン別の影響をまとめると、次のとおりです。

| 利用パターン | 旧プラン | 新プラン | 影響 |
| --- | --- | --- | --- |
| ライトユーザー中心 | バンドル内で完結 | 使った分だけ請求 | コスト増 |
| ヘビー API 利用 | オーバーエージ主体 | ほぼ同じ構造 | ほぼ中立 |
| Claude Code エージェント常用 | バンドル超過が常態 | 全消費が API 標準レートで請求 | 可視性向上、ピーク時要注意 |
| バンドル枠の余りに依存 | 余剰トークンを共有 | 余剰の概念が消える | 設計見直し必須 |

## API 標準レートとの整合

新モデルの「API 標準レート」とは、Anthropic の公開している Messages API の従量レートと同一です。Opus 4.7、Sonnet 4.7、Haiku など各モデルの 1M トークンあたりレートに、入力・出力それぞれの消費を掛けた額がそのまま請求されます。

エンタープライズ側がコスト管理を行う上で重要なのは、次の点です。

- **シート料金とトークン料金は別物**: 月次予算を作るなら、それぞれを独立した支出項目として計上する
- **モデル選択がコストを大きく左右する**: Opus 4.7 と Haiku ではトークン単価が大きく異なる
- **プロンプトキャッシュ・バッチ処理を活用すべき**: API 標準レートのまま走らせると重い処理ほどコストが効く
- **ワークスペース単位でレート制限とコストを観測する**: Rate Limits API や Analytics API を活用すれば、人やチーム単位の使用量を把握しやすい

特にプロンプトキャッシュは、エージェント系ワークロードで効果が大きく、有効化するだけで実質コストが数十パーセント下がるケースもあります。新モデルではキャッシュヒットも従量課金の対象（ただし通常レートより安価）になるため、**キャッシュ設計はコスト管理の中心的なテーマ**になります。

## グランドファザード契約の取り扱い

旧プランで複数年契約を結んでいる組織については、契約満了まで現行条件が維持される場合があります。ただし、次の点に注意が必要です。

- **契約更新タイミングで強制的に新モデルに移行**: バンドル枠を維持できる例外契約は限定的
- **モデルアップデートに伴う消費増は旧契約でも反映される**: 旧契約のバンドル枠を超過する分は API 標準レートで請求
- **オプションサービス（Claude Code、Cowork など）は契約後追加分のみ新モデル準拠**: 既存サービスは旧条件、追加サービスは新条件、というハイブリッドが起きうる

このため、**自社の契約更新時期を正確に把握**し、その3〜6ヶ月前から FinOps 観点で予算再計画を始めることが推奨されます。

## 何を意思決定すべきか — エンタープライズ側のチェックリスト

新モデル下での意思決定は、概ね次のステップで進めるとよいでしょう。

```
[1] 現行契約の確認
    └ 残存期間 / バンドル枠 / 実績消費 / オーバーエージ比率

[2] 利用実態の棚卸し
    └ シート保有者 vs アクティブユーザー
    └ Claude / Claude Code / Cowork それぞれの消費
    └ モデル別（Opus / Sonnet / Haiku）の消費比率

[3] 新モデル下のシミュレーション
    └ シート単価 × 人数
    └ 月次トークン消費 × API 標準レート
    └ プロンプトキャッシュ / バッチ適用後の実効レート

[4] 契約交渉項目の整理
    └ コミット契約（年間消費コミット）の交渉余地
    └ 段階値引きの有無
    └ ワークスペース単位のクォータ設計

[5] 内部統制の準備
    └ ワークスペース単位での予算管理
    └ ユーザー単位の使用量モニタリング
    └ 異常検知（ランナウェイエージェント対策）
```

特に重要なのは、Step 5 の「内部統制」です。従量課金モデルでは、設定ミスや暴走したエージェントが**短期間で多額のトークンを消費する**リスクが顕在化します。Anthropic の Rate Limits API や Analytics API を組み合わせ、リアルタイムに使用量を監視するパイプラインを準備しておくと、こうしたリスクを大幅に低減できます。

## なぜ Anthropic は今このタイミングで動いたのか

報道情報を踏まえると、移行の背景として複数の要因が指摘されています。

- **ランレートの急成長**: 2025年末の年間ランレート約90億ドルから、2026年4月には約300億ドルへ。4ヶ月で約3.3倍という伸びは、固定収益型プランでは追従しきれない
- **キャパシティ制約**: GPU/TPU リソースの逼迫により、固定枠を約束し続けるリスクが高まった
- **ヘビーユーザー集中**: 1,000社以上が年間100万ドル以上を支払う構造になり、フラットフィーよりも変動課金のほうが収益管理に適している
- **業界全体のトレンド**: OpenAI は2026年4月に Codex を従量制へ、GitHub も Copilot を制限強化、Windsurf もデイリークォータ制へ。Anthropic だけがフラットフィーを維持する合理性は乏しい
- **稼働率の課題**: 90日平均でAPI稼働率が約99%という水準は、エンタープライズグレードの99.99%とはギャップがあり、料金体系に「使った分だけ払う」性格を持たせることでサービスレベル議論との整合をとりやすい

つまり、**料金体系の見直しは、Anthropic 自身のビジネスモデル安定化と、エンタープライズ顧客のリスク受容のリバランス**でもあると言えます。

## 業界全体の流れ — フラットフィー時代の終焉

Anthropic だけでなく、生成 AI 各社が同時期に従量制へ舵を切っています。

| ベンダー | 製品 | 変更内容 | 時期 |
| --- | --- | --- | --- |
| OpenAI | Codex | フラットから従量制へ | 2026年4月 |
| GitHub | Copilot | 月間上限を厳格化 | 2026年4月10日 |
| Windsurf | IDE | クレジット制から日次クォータ制へ | 2026年3月 |
| Anthropic | Claude Enterprise | バンドルトークン廃止 | 2026年4月（11月開始） |

共通する流れは、次の3点に集約できます。

1. **モデル消費の増加に固定フィーが追従できない**
2. **重量ユーザーと軽量ユーザーの利用差が拡大した**
3. **キャパシティ制約により「無制限」を約束できなくなった**

この変化は、ユーザー側にとっては「自由度の高い設計が可能」「使った分だけ払えばよい」という側面と、「予算化が難しくなる」「個人の使用量管理が必要」という側面の両方をもたらします。

## まとめ — 新しい契約観の準備

Anthropic の料金体系移行は、単なる値上げ・値下げの議論ではなく、**「AI 利用を電気・通信のような従量サービスとして扱う」**という構造的シフトです。

エンタープライズ側に求められる準備は、おおむね次のとおりです。

- 自社の利用実態を**シート数・トークン消費・モデル別**にブレークダウンする
- 契約更新時期と新モデル適用タイミングをカレンダー管理する
- ワークスペース単位での**予算割当・モニタリング・異常検知**の体制を整える
- プロンプトキャッシュ・バッチ・モデル選択を**コスト最適化のレバー**として活用する
- 突発的な消費増に備え、**Rate Limits API などによる自動ガード**を組む

短期的にはコスト構造の見直しを迫られますが、中長期では「使った分だけ払う」モデルのほうが、AI 活用の幅と深さを正確に支出に反映できる、という利点もあります。フラットフィー時代の終わりは、AI を本格的に経営インフラとして組み込むための、新しい契約観の始まりでもあると言えるでしょう。
