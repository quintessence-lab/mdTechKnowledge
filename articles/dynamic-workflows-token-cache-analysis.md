> ## 要点
>
> - Claude Code の **Dynamic Workflows を15回実行**して採取したトークン実測（**個別の用途名は伏せ、数値のみ**）を分析。
> - **キャッシュ読込比率（読込トークン÷合算）は 44〜90%**、**キャッシュによるコスト削減率は 23〜77%** に分布。**両者はほぼ連動**して動く。
> - 最大の傾向は **「並列度（サブエージェント数）が高い実行ほど、キャッシュ読込比率もコスト削減率も高い」**。5〜6体規模は比率・削減率とも低め、31〜52体規模は高め。
> - 理由は明快で、**多数のサブエージェントが同じ共有プレフィックス（システムプロンプト・ツール定義・親の指示・共有コンテキスト）を再利用**するため。キャッシュ読込はキャッシュ無し換算の約1/10単価で済む。
> - **1エージェント当たりコストは規模を上げても膨らまない**（31〜52体で $0.68〜$0.98）。大規模ファンアウトはキャッシュ上で費用効率的。

## データの前提

本分析は、Dynamic Workflows を実運用で15回回した際のトークンログ（ローカル正本）に基づきます。**どの実行が何の用途だったかは伏せ、純粋に数値だけ**を扱います。

- **計測軸**：各サブエージェントのトランスクリプトを「生入力／キャッシュ作成／キャッシュ読込／出力」の4区分で積算した値（＝トランスクリプト合算）。
- **概算コスト**：Opus 4.8 の標準価格での**試算**（入力 $5 / キャッシュ作成 $6.25 / キャッシュ読込 $0.5 / 出力 $25、いずれも per MTok）。実課金額そのものではありません。
- 期間は2026年5月下旬〜6月中旬。エージェント数（並列度）は5〜52体と幅があります。

### 指標の定義

| 指標 | 定義 |
|---|---|
| **キャッシュ読込比率** | キャッシュ読込トークン ÷ トランスクリプト合算。多数のサブエージェントが共有文脈をどれだけ再利用したかの度合い。 |
| **キャッシュ無し仮想コスト** | 「生入力＋作成＋読込」を**すべて生入力単価（$5）**で課金し、出力を加えた額。キャッシュが無かった場合の最悪コスト。 |
| **トークン削減率** | （仮想コスト − 実コスト）÷ 仮想コスト。キャッシュ機構が圧縮したコストの割合（高いほど効率的）。 |
| **1エージェント当たりコスト** | 実コスト ÷ サブエージェント数。 |

## 分析① キャッシュ読込比率の傾向

キャッシュ読込比率は **44.3%〜90.1%** に分布し、多くは **58〜85%** に収まります。

- **下限側（44%前後）**：5体の執筆中心ジョブと、12体でも出力が極端に軽い要約ジョブ。**生成（出力）主体で、共有文脈の再利用が少ない**実行ほど比率が低い。
- **上限側（79〜90%）**：31〜52体の大規模並列ジョブ。**共有プレフィックスを多数のサブエージェントが繰り返し読む**ため、合算トークンの大半がキャッシュ読込で占められる。

つまりキャッシュ読込比率は「**そのワークフローがどれだけ共有文脈を使い回したか**」の指標として読めます。

## 分析② トークン削減率はキャッシュ読込比率と連動する

トークン削減率は **23.4%〜77.1%** に分布し、**キャッシュ読込比率とほぼ同じ順序**で並びます。

- 読込比率44%級 → 削減率23〜25%（最下位グループ）
- 読込比率73〜79%級 → 削減率58〜66%
- 読込比率85〜90%級 → 削減率72〜77%（最上位）

これは定義上当然の連動です。キャッシュ読込は**キャッシュ無し換算（生入力単価 $5）の約1/10（$0.5）**で課金されるため、合算に占める読込比率が上がるほど、「キャッシュが無かった場合の仮想コスト」との差＝削減額が大きくなります。**キャッシュ読込比率が高い ≒ コスト削減率が高い**、と覚えておけば実用上十分です。

## 分析③ 核心：並列度が上がるほど効率が上がる

最も重要な傾向は、**サブエージェント数（並列度）とキャッシュ効率の正の相関**です。

![サブエージェント数（横軸）とキャッシュ読込比率・トークン削減率（縦軸）の散布図。並列度が高いほど両指標が上がる](https://quintessence-lab.github.io/mdTechKnowledge/images/dwf-token/parallelism-vs-ratios.svg)

*図2: 横軸がサブエージェント数（並列度）、縦軸が割合(%)。キャッシュ読込比率（青）とトークン削減率（赤）はいずれも並列度とともに上昇する（破線はそれぞれの回帰直線）。5〜6体規模は下方にばらつき、31〜52体規模は上方に集まる。*

- **5〜6体規模**：読込比率44〜73%、削減率23〜58%（ばらつき大、下限側に寄る）
- **8〜18体規模**：読込比率58〜83%、削減率40〜68%（中位）
- **31〜52体規模**：読込比率79〜90%、削減率65〜77%（上位に集中）

**なぜ並列度が上がると効率が上がるのか**：Dynamic Workflows では、各サブエージェントが「システムプロンプト・全ツール定義・親エージェントの指示・共有コンテキスト」という**大きな共通プレフィックス**を抱えて起動します。このプレフィックスは**一度キャッシュに書き込めば、ファンアウトした全サブエージェントが読込で再利用**できます。並列度が高いほど「1回の書き込み × 多数の読込」という構図が強まり、読込比率＝削減率が押し上がる——というメカニズムです。

逆に、エージェント数が少なく、かつ**新規の出力（生成）が主体**の実行（執筆・軽量要約）は、共有文脈の再利用余地が小さく、効率は相対的に低くなります。**「単なるエージェント数」よりも「共有プレフィックスをどれだけ読み回すか」**が本質である点に注意してください（12体でも軽量要約は44%にとどまる）。

## 分析④ 1エージェント当たりコストと規模効率

「並列度を上げるとコストが線形以上に膨らむのでは」という直感に反し、**1体当たりコストは規模を上げても膨らみません**。

- 31体：**$0.68 / $0.71**、52体：**$0.98** ——いずれも全体の中で低位。
- 一方、8体でも監査・出力が重い実行は **$2.54** と高い。

これは、規模が大きいほどキャッシュ再利用が効くため、**サブエージェントを増やしても1体当たりの実コストはむしろ抑えられる**ことを示します。コストを左右するのは**エージェント数そのものより、出力量とキャッシュ再利用度**です。大規模ファンアウトは「キャッシュ機構の上では費用効率的」と言えます。

## 外れ値と限界

- **軽量要約（12体・44.3%・削減25%）**：エージェント数の割に読込比率・削減率が低い外れ値。出力が極端に軽く、共有文脈の再利用も少なかったため。並列度と効率の相関は「傾向」であって厳密な線形ではない。
- **監査込み（8体・$2.54/体）**：出力トークンが突出して多く、1体当たりコストを押し上げた例。
- 本データは **n=15** の実測サンプルであり、用途・フェーズ構成が不均一。**厳密な統計的因果ではなく傾向の記述**として読むこと。
- コストは Opus 4.8 標準価格での**試算**で、実課金とは異なる。「ハーネス報告のトークン」と「トランスクリプト合算」は計測軸が違うため、コスト感は概算コスト列で見るのが実態に近い。

## まとめ

- **キャッシュ読込比率（44〜90%）とトークン削減率（23〜77%）はほぼ連動**する。読込比率が高い実行＝削減率が高い実行。
- **並列度（サブエージェント数）が高いほど、両指標は高くなる**。共有プレフィックスを多数のサブエージェントが読み回すためで、これが Dynamic Workflows のコスト効率の源泉。
- **1体当たりコストは規模で膨らまない**。費用を決めるのは「エージェント数」より「出力量×キャッシュ再利用度」。
- 実務的な示唆：**共有コンテキストを厚くして多数に並列展開する設計**ほどキャッシュが効き、単位コストが下がる。一方、出力主体・小規模・使い捨て文脈の実行はキャッシュ効率が出にくい。

> 本記事は、Dynamic Workflows の自前実測ログ（15実行・匿名集計）に基づく数値分析です。数値はすべて内部計測値であり、コストは Opus 4.8 標準価格での試算です。
